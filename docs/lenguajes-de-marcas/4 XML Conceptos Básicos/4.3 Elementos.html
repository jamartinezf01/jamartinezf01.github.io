4.3 Elementos XML

Un elemento XML se define con una etiqueta de apertura, un contenido y una etiqueta de cierre.

<pokemon>
    <nombre>Pikachu</nombre>
</pokemon>

Aquí, <pokemon> es el elemento principal y <nombre> es un subelemento.

Contenido de un Elemento​

Los elementos en XML representan la estructura principal de los datos y pueden contener:

  1. Texto
  2. Otros elementos anidados
  3. Atributos
elementos vacios

Un Elemento puede estar vacío siempre que tenga etiquetas de apertura y cierre. Un elemento vacío puede tener atributos.

1. Elementos con Texto​

Un elemento con texto contiene solo datos sin subelementos ni atributos.

  • Ejemplo
<nombre>Pikachu</nombre>

Aquí, el texto "Pikachu" está contenido dentro del elemento <nombre>.

2. Elementos con Otros Elementos Anidados​

Un elemento anidado contiene otros elementos en su interior.

  • Ejemplo
<pokemon>
    <nombre>Pikachu</nombre>
    <tipo>Eléctrico</tipo>
</pokemon>
  • <pokemon> es el elemento principal.
  • <nombre> y <tipo> son subelementos.

3. Elementos con Atributos​

Un elemento con atributos almacena información adicional dentro de la etiqueta de apertura.

  • Ejemplo
<pokemon id="025">
    <nombre>Pikachu</nombre>
    <tipo>Eléctrico</tipo>
</pokemon>

Aquí, id="025" es un atributo que identifica a Pikachu.

Diferencia entre Elementos y Atributos 📌
  • ✅ Elementos: Usar cuando los datos son estructurados y pueden contener más información.
  • ✅ Atributos: Usar para metadatos o identificadores.

Ejemplo alternativo con más datos usando elementos en lugar de atributos:

<pokemon>
    <id>025</id>
    <nombre>Pikachu</nombre>
    <tipo>Eléctrico</tipo>
</pokemon>

Ambas formas son correctas, pero la elección depende del contexto.

Nombres de elementos​

Reglas para Nombrar Elementos​

Los nombres de los elementos siguen reglas estrictas:

1. Deben comenzar con una letra o un guion bajo (_)​

correcto
<Pokemon>...</Pokemon>
<_habilidad>...</_habilidad>

2. No pueden iniciar con "xml" en ninguna combinación de mayúsculas o minúsculas​

Incorrecto:
<xmlDatos>...</xmlDatos>

3. No pueden contener espacios en blanco​

Incorrecto:
<nombre completo>Pikachu</nombre completo>
Correcto
<nombre_completo>Pikachu</nombre_completo>

4. No pueden incluir caracteres especiales como &, <, >, ", '​

Incorrecto:
<tipo&habilidad>Eléctrico</tipo&habilidad>

Buenas Prácticas para Nombrar Elementos​

Para mejorar la legibilidad y mantenimiento del XML, se recomienda seguir estas buenas prácticas:

1. Usar nombres descriptivos​

Correcto:
<nombre>Pikachu</nombre>
Incorrecto:
<n>Pikachu</n>

2. Evitar nombres genéricos​

Incorrecto:
<dato>Pikachu</dato>
Correcto:
<nombre>Pikachu</nombre>

3. Ser coherente con las convenciones de nombres​

Incorrecto: (nombres mezclados):
<nombreCompleto>Pikachu</nombreCompleto>
<nombre_completo>Charmander</nombre_completo>
Correcto: (seguir una convención consistente):
<nombre_completo>Pikachu</nombre_completo>
<nombre_completo>Charmander</nombre_completo>

4. Usar singular o plural según corresponda​

Incorrecto:
<pokemons>
    <pokemon>Pikachu</pokemon>
    <pokemon>Charmander</pokemon>
</pokemons>
Correcto:
<pokedex>
    <pokemon>
        <nombre>Pikachu</nombre>
    </pokemon>
    <pokemon>
        <nombre>Charmander</nombre>
    </pokemon>
</pokedex>

5. Evitar el uso de guiones (-) en nombres de elementos​

  • Motivo: Algunos programas pueden interpretar el guion como una operación matemática de resta.
Incorrecto:
<tipo-primario>Fuego</tipo-primario>
Correcto:
<tipo_primario>Fuego</tipo_primario>

6. Evitar el uso del punto (.) en nombres de elementos​

  • Motivo: Algunos sistemas pueden interpretar el punto como una propiedad de objeto.
Incorrecto:
<pokemon.nombre>Pikachu</pokemon.nombre>
Correcto:
<nombre_pokemon>Pikachu</nombre_pokemon>

7. Evitar el uso de dos puntos (:) en nombres de elementos​

  • Motivo: XML reserva los dos puntos para los espacios de nombres.
Incorrecto:
<pokemon:tipo>Eléctrico</pokemon:tipo>
Correcto:
<tipo_pokemon>Eléctrico</tipo_pokemon>

8. Cuidado con caracteres no ingleses (é, ò, á, ñ)​

  • Motivo: Aunque son válidos en XML, pueden causar problemas si el software no los admite correctamente.
Ejemplo con posible problema:
<título>Pikachu</título>
Alternativa segura:
<titulo>Pikachu</titulo>
  • Solución recomendada: Asegurarse de que la codificación del documento sea UTF-8, especificándolo en la declaración XML: 
    <?xml version="1.0" encoding="UTF-8"?>

Ejemplo​

<?xml version="1.0" encoding="UTF-8"?>
<!-- Pokedex con nombres de elementos siguiendo buenas prácticas -->
<pokedex>
    <pokemon id="001">
        <nombre_pokemon>Bulbasaur</nombre_pokemon>
        <tipo_primario>Planta</tipo_primario>
        <tipo_secundario>Veneno</tipo_secundario>
        <habilidades>
            <habilidad>Espesura</habilidad>
            <habilidad_oculta>Clorofila</habilidad_oculta>
        </habilidades>
       <descripcion>
            <![CDATA[Bulbasaur < Ivysaur < Venusaur en evolución.]]>
        </descripcion>
    </pokemon>

    <pokemon id="004">
        <nombre_pokemon>Charmander</nombre_pokemon>
        <tipo_primario>Fuego</tipo_primario>
        <habilidades>
            <habilidad>Mar Llamas</habilidad>
            <habilidad_oculta>Poder Solar</habilidad_oculta>
        </habilidades>
        <descripcion>
            <![CDATA[Charmander < Charmeleon < Charizard en evolución.]]>
        </descripcion>
    </pokemon>

    <pokemon id="025">
        <nombre_pokemon>Pikachu</nombre_pokemon>
        <tipo_primario>Eléctrico</tipo_primario>
        <habilidades>
            <habilidad>Electricidad Estática</habilidad>
            <habilidad_oculta>Pararrayos</habilidad_oculta>
        </habilidades>
        <descripcion>
            <![CDATA[Pikachu < Raichu en evolución.]]>
        </descripcion>
    </pokemon>
</pokedex>

Recuerda​

Recuerda...

Los elementos en XML son la base de la estructura de datos. Para garantizar un documento bien formado:

✅ Usa nombres descriptivos y evita abreviaciones sin contexto.
✅ No utilices caracteres que puedan generar errores en ciertos sistemas (-, ., :).
✅ Mantén coherencia en el estilo de nombres (guion bajo o camelCase, pero no mezclados).
✅ Asegúrate de que el software pueda manejar caracteres especiales si los usas.

Siguiendo estas recomendaciones, los documentos XML serán más compatibles, organizados y fáciles de procesar en múltiples plataformas. 🚀🔥
