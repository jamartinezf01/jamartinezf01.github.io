4.4 Atributos XML

Los atributos en XML proporcionan información adicional sobre un elemento y se declaran dentro de la etiqueta de apertura. Son utilizados para describir características del elemento al que pertenecen, como identificadores, idiomas o metadatos.

Los atributos permiten agregar información relevante sin necesidad de crear más elementos.

<pokemon id="025" tipo_primario="Eléctrico">
    <nombre>Pikachu</nombre>
    <habilidad principal="true">Electricidad Estática</habilidad>
</pokemon>
info
  • id="025": Identifica de forma única a Pikachu dentro del documento.
  • tipo_primario="Eléctrico": Define el tipo del Pokémon.
  • principal="true" en <habilidad> indica que es la habilidad principal.

Cuándo Usar Atributos en XML​

Los atributos son útiles cuando la información es descriptiva y no requiere ser estructurada en elementos secundarios.

📌 Usa atributos cuando:
✅ Se necesita una identificación única (id="...").
✅ La información es breve y no necesita formato (idioma="español").
✅ Los datos son metadatos del elemento (tipo_primario="Fuego").

📌 Usa elementos cuando:
✅ La información puede repetirse (varios tipos, varias habilidades).
✅ Los datos necesitan anidación o estructura compleja.
✅ Se requiere formatear el contenido (como descripciones largas).

En General

✅ Si quieres que los datos almacenados en el XML sean más legibles y estructurados utiliza más elementos que atributos si quieres que sea más abreviado y menos legible utiliza "más atributos".
✅ Los elementos se usan para datos importantes que necesitan estructura y manipulación.
✅ Los atributos almacenan metadatos o identificadores, información breve y no estructurada.

Recomendaciones de Atributo vs. Elemento​

1. Si el dato es un identificador​

ID, código único, DNI, serial, etc., deben ser atributos. En bases de datos relacionales, los identificadores son claves primarias y deben representarse en XML como atributos.

Ejemplo Incorrecto (ID como elemento):
<pokemon>
    <id>025</id>
    <nombre>Pikachu</nombre>
</pokemon>
Ejemplo Correcto:
<pokemon id="025">
    <nombre>Pikachu</nombre>
</pokemon>
  • Motivo: Los identificadores deben ser metadatos y no parte de la estructura de datos.

2. Si el dato es importante​

Los datos clave deben representarse con elementos, no atributos. Si un dato es fundamental para la estructura del documento, debe ser un elemento.

Ejemplo Incorrecto (usar atributos para datos importantes):
<pokemon nombre="Charizard" tipo1="Fuego" tipo2="Volador" />
Ejemplo Correcto (usando elementos para datos importantes):
<pokemon>
    <nombre>Charizard</nombre>
    <tipo>Fuego</tipo>
    <tipo>Volador</tipo>
</pokemon>
  • Motivo: Los tipos pueden cambiar o aumentar en el futuro, por lo que es mejor representarlos como elementos.

3. Si el dato se puede subdividir en el futuro​

Los atributos no se pueden subdividir en el futuro, si tenemos un XML "extenso“, y en el futuro pensamos que se puede subdividir ese dato, es más rápido hacerlo desde un elemento que desde un atributo, ya que no tengo que reestructurar "tanto" el documento.

Incorrecto (atributo innecesario):
<pokemon id="004" tipos="Fuego">
    <nombre>Charmander</nombre>
</pokemon>
Correcto (usar elementos en vez de atributos):
<pokemon id="004">
    <nombre>Charmander</nombre>
    <tipo>Fuego</tipo>
</pokemon>
  • Motivo: Si un Pokémon puede tener más de un tipo (Fuego, Volador), usar elementos facilita la gestión de múltiples valores.

4. Si un dato depende de otro​

Si un elemento está estrechamente unido con otro, es decir que si modifico uno se modifica el otro no se aconseja ponerlo cómo atributos. Si sabemos que la unidad va a modificar la cantidad, es decir no es lo mismo 250ml que 250g se aconseja como elemento uno y el otro que modifica como atributo

Incorrecto
<cantidad>
    <valor>250</valor>
    <unidad>ml</unidad>
</cantidad>
Correcto (unidades como atributo)
<cantidad unidad="ml">250</cantidad>
  • Motivo: Cambiando la unidad de medida se modifica el significado del elemento.

5. El problema de las fechas​

Un atributo no se puede dividirse "contener unidades más pequeñas". Aquí viene la típica duda de la fecha. ¿La pongo como atributo (YYYY/MM/DD)?, ¿la pongo como un único elemento (YYYY/MM/DD)? o ¿la pongo como un elemento fecha y tres subelementos (día, mes, año)?.
Es aconsejable la tercera opción y tiene una razón. Nos tenemos que preguntar ¿Qué queremos hacer con la fecha?. ¿Queremos solo mostrarla? las tres opciones son correctas. ¿Queremos manipular por separado con alguna sentencia XQUERY los días, meses o años?, la tercera opción es la correcta.

Solo para mostrar
<pokemon fecha_captura="2025-03-15">
    <nombre>Squirtle</nombre>
</pokemon>
Alternativa multipropósto (fecha dividida en elementos):
<fecha>
    <dia>15</dia>
    <mes>03</mes>
    <año>2025</año>
</fecha>
Motivo

Si en el futuro necesitamos filtrar Pokémon por mes o año, la versión con elementos será más fácil de manejar.

Formato

Por norma general cuando usemos atributos para almacenar fechas emplearemos el formato ISO-8601 Se conoce también como formato americano YYYY-MM-DD. Si es necesario incluit la hota YYY-MM-DDThh:mm:ss.

Atributos Predefinidos en XML​

XML tiene algunos atributos con significados especiales que son ampliamente utilizados.

1. id (Identificador Único)​

El atributo id se usa para identificar de manera única un elemento dentro del documento.

<pokemon id="001">
    <nombre>Bulbasaur</nombre>
</pokemon>
Reglas para id:
  • Debe ser único dentro del documento.
  • No puede repetirse en otros elementos.

2. lang (Definir el Idioma)​

El atributo lang indica el idioma en el que está escrito el contenido del elemento.

<pokemon id="150" lang="es">
    <nombre>Mewtwo</nombre>
    <descripcion>Mewtwo es un Pokémon legendario creado a partir del ADN de Mew.</descripcion>
</pokemon>
Usos de lang:
  • Permite indicar que el contenido está en un idioma específico.
  • Puede usarse en elementos individuales o en toda la estructura XML.

3. xmlns (Espacios de Nombres en XML)​

El atributo xmlns define un espacio de nombres que evita conflictos cuando diferentes XML utilizan los mismos nombres de elementos.

<pokedex xmlns="http://ejemplo.com/pokedex">
    <pokemon id="007">
        <nombre>Squirtle</nombre>
        <tipo>Agua</tipo>
    </pokemon>
</pokedex>
Importancia de xmlns:
  • Evita conflictos de nombres en documentos XML combinados.
  • Permite que el XML sea interpretado correctamente en distintos sistemas.
