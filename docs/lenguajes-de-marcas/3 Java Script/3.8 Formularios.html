3.8 Formularios

Los formularios son una parte esencial de cualquier aplicación web, porque conectan la interfaz con los datos del usuario. En frontend, casi todo lo relacionado con formularios se resume en 4 ideas:

  1. Acceder al formulario y a sus campos
  2. Leer y escribir valores (texto, checkbox, select, etc.)
  3. Escuchar eventos (submit, input, change) para reaccionar a lo que hace el usuario
  4. Validar antes de enviar, usando validación nativa y, si hace falta, lógica propia

Aunque se validen datos en el navegador, nunca se debe confiar en la validación del cliente si esos datos van a un servidor. En este tema se trabaja solo la parte web.

1️⃣ Estructura básica de un formulario​

Un formulario en HTML se define con <form> y suele incluir inputs y un botón de envío. Lo más importante para trabajar bien con JS es:

  • id para seleccionar el formulario fácilmente
  • name en los campos, porque es clave para form.elements, FormData y envíos
  • label asociado con for para accesibilidad y clic cómodo
Formulario básico (HTML)
<form id="formulario-registro">
  <label for="usuario">Usuario</label>
  <input id="usuario" name="usuario" type="text" required />

  <label for="correo">Correo</label>
  <input id="correo" name="correo" type="email" required />

  <button type="submit">Enviar</button>
</form>
Recomendación

En proyectos reales conviene evitar ñ en name y en id Por ejemplo, contrasena en lugar de contraseña

2️⃣ Acceso a formularios y campos​

Hay dos enfoques habituales para acceder a formularios y sus elementos:

  • Selección directa con querySelector (muy común hoy)
  • Colecciones del DOM con document.forms y form.elements (muy útil y muy visto en manuales)

🟩 Selección directa con querySelector y getElementById​

Es el enfoque más directo: seleccionas el formulario y luego los campos que te interesan.

Selección directa
const formulario = document.querySelector("#formulario-registro")
const inputUsuario = document.querySelector("#usuario")
const inputCorreo = document.querySelector("#correo")
Idea clave

Este enfoque es perfecto cuando tienes pocos campos o IDs claros Si el formulario crece, suele ser más ordenado trabajar con form.elements

🟨 Acceso con document.forms​

document.forms devuelve una colección con todos los formularios del documento. Se puede acceder por:

  • índice document.forms[0]
  • nombre document.forms["miFormulario"] si el formulario tiene name
Acceso con document.forms
<form id="formulario1" name="miFormulario">
  <input type="text" name="usuario" placeholder="Usuario" />
  <input type="password" name="contrasena" placeholder="Contraseña" />
</form>

<script>
  const formulario = document.forms["miFormulario"]
  console.log(formulario.id) // "formulario1"
</script>

🟧 Acceso con form.elements​

form.elements devuelve una colección con todos los campos del formulario (inputs, selects, botones). Lo más típico es acceder por name.

Acceso con form.elements
<form id="formulario">
  <input type="text" name="usuario" placeholder="Usuario" />
  <input type="password" name="contrasena" placeholder="Contraseña" />
</form>

<script>
  const formulario = document.getElementById("formulario")

  const usuario = formulario.elements["usuario"]
  const contrasena = formulario.elements["contrasena"]

  console.log(usuario.placeholder)   // "Usuario"
  console.log(contrasena.type)       // "password"
</script>
Regla rápida
  • Si ya tienes el formulario, form.elements["..."] es muy cómodo
  • Para que funcione bien, pon siempre name en los campos

3️⃣ Leer y escribir valores en los campos​

En frontend, la mayor parte del trabajo con formularios es leer valores, validarlos y (a veces) escribirlos.

🟩 value (inputs, textarea, select)​

value se usa en la mayoría de campos: texto, email, password, textarea, select.

Leer y escribir value
const usuario = formulario.elements["usuario"]

console.log(usuario.value)   // lo que haya escrito el usuario
usuario.value = "NuevoUsuario" // cambia el valor desde JS
Caso típico

Se usa para rellenar un formulario al editar datos, o para limpiar campos tras enviar

🟨 checked (checkbox y radio)​

Los checkbox y radio se controlan con checked.

Checkbox y radio
<input type="checkbox" id="aceptar" name="aceptar" />
<label for="aceptar">Acepto los términos</label>

<input type="radio" name="rol" value="user" checked /> Usuario
<input type="radio" name="rol" value="admin" /> Admin
checked
const aceptar = formulario.elements["aceptar"]

console.log(aceptar.checked) // true o false
aceptar.checked = true       // marcar desde JS
Ojo con radio

Con radios, form.elements["rol"] puede devolverte una colección Lo más práctico es leer el seleccionado con un selector

Leer el radio seleccionado
const rol = formulario.querySelector('input[name="rol"]:checked')
console.log(rol ? rol.value : null)

🟦 Leer archivos (file input)​

Cuando un input es type="file", el valor real se obtiene desde files.

File input
<input type="file" id="foto" name="foto" accept="image/*" />
Leer archivo seleccionado
const foto = formulario.elements["foto"]

if (foto.files.length > 0) {
  const archivo = foto.files[0]
  console.log(archivo.name)
  console.log(archivo.type)
  console.log(archivo.size)
}

4️⃣ Eventos en formularios​

Los formularios generan eventos muy útiles. Los más comunes son:

  • submit: al enviar el formulario
  • input: cada vez que el usuario modifica el valor
  • change: cuando cambia y pierde foco (o confirma un cambio)

🟩 Evento submit​

Se dispara cuando se intenta enviar el formulario. Normalmente se usa para:

  • interceptar el envío con preventDefault()
  • validar
  • construir datos (JSON o FormData)
  • enviar con fetch (si aplica)
submit básico
formulario.addEventListener("submit", (evento) => {
  evento.preventDefault()
  console.log("Formulario enviado")
})

🟨 Evento input​

Se dispara en cada cambio inmediato. Es ideal para:

  • validación en vivo
  • contadores de caracteres
  • previsualizaciones
input
const usuario = formulario.elements["usuario"]

usuario.addEventListener("input", () => {
  console.log("Escribiendo:", usuario.value)
})

🟧 Evento change​

Se dispara cuando el valor cambia y se confirma el cambio (normalmente al perder foco).

change
const aceptar = formulario.elements["aceptar"]

aceptar.addEventListener("change", (evento) => {
  console.log(`Checkbox: ${evento.target.checked ? "Marcado" : "Desmarcado"}`)
})

🟥 Ejemplo completo combinando eventos​

Formulario completo
<form id="formulario">
  <input type="text" name="usuario" placeholder="Usuario" required />
  <input type="password" name="contrasena" placeholder="Contraseña" required />

  <input type="checkbox" name="aceptar" id="aceptar" />
  <label for="aceptar">Acepto los términos</label>

  <button type="submit">Enviar</button>
</form>

<script>
  const formulario = document.getElementById("formulario")

  formulario.addEventListener("submit", (evento) => {
    evento.preventDefault()

    const usuario = formulario.elements["usuario"]
    const contrasena = formulario.elements["contrasena"]
    const aceptar = formulario.elements["aceptar"]

    if (!usuario.value || !contrasena.value || !aceptar.checked) {
      alert("Por favor, completa todos los campos y acepta los términos.")
      return
    }

    console.log("Formulario válido. Enviando...")
  })

  formulario.elements["usuario"].addEventListener("input", () => {
    console.log("El usuario está escribiendo...")
  })

  formulario.elements["aceptar"].addEventListener("change", (evento) => {
    console.log(`Checkbox: ${evento.target.checked ? "Marcado" : "Desmarcado"}`)
  })
</script>

5️⃣ Validación de formularios​

La validación puede hacerse de dos formas que suelen combinarse:

  • Validación nativa HTML: required, minlength, type="email", pattern, etc.
  • Validación en JS: reglas propias y mensajes personalizados

🟩 Validación nativa con checkValidity() y reportValidity()​

checkValidity() devuelve true o false según las restricciones HTML. reportValidity() además muestra el mensaje del navegador.

checkValidity y reportValidity
const correo = formulario.elements["correo"]

if (!correo.checkValidity()) {
  correo.reportValidity()
}

🟨 Mensajes personalizados con setCustomValidity()​

setCustomValidity() permite definir un mensaje de error propio.

  • si le pasas un texto, el campo pasa a ser inválido
  • si le pasas "", limpias el error y vuelve a validar normalmente
Validación personalizada (email)
<form id="formulario">
  <input type="email" name="correo" placeholder="Correo electrónico" required />
  <button type="submit">Enviar</button>
</form>

<script>
  const formulario = document.getElementById("formulario")

  formulario.addEventListener("submit", (evento) => {
    evento.preventDefault()

    const correo = formulario.elements["correo"]

    // Limpia mensajes anteriores
    correo.setCustomValidity("")

    if (!correo.checkValidity()) {
      correo.setCustomValidity("Por favor, introduce un correo válido.")
      correo.reportValidity()
      return
    }

    console.log("Formulario válido. Enviando...")
  })
</script>
Importante

Si usas setCustomValidity, acuérdate de limpiar con setCustomValidity("") Si no, el campo puede quedarse “atascado” como inválido

🟧 Estados de validación: validity y validationMessage​

Cuando algo falla, puedes inspeccionar exactamente por qué.

validity y validationMessage
const correo = formulario.elements["correo"]

console.log(correo.validity)           // objeto con estados
console.log(correo.validationMessage)  // mensaje actual del navegador

🟥 Validación manual sencilla (reglas propias)​

Aunque exista validación nativa, a veces quieres reglas tuyas.

Validación manual simple
function validarUsuario(valor) {
  return valor.trim().length >= 3
}

formulario.addEventListener("submit", (evento) => {
  evento.preventDefault()

  const usuario = formulario.elements["usuario"]

  if (!validarUsuario(usuario.value)) {
    alert("El usuario debe tener al menos 3 caracteres")
    return
  }

  console.log("OK")
})

6️⃣ Recoger datos del formulario​

🟩 Recoger valores “a mano”​

Es directo y claro cuando son pocos campos.

Recoger datos manualmente
const datos = {
  usuario: formulario.elements["usuario"].value,
  correo: formulario.elements["correo"].value,
  aceptar: formulario.elements["aceptar"].checked
}

console.log(datos)

🟨 FormData (muy útil para formularios grandes y archivos)​

FormData lee automáticamente los campos con name y permite incluir archivos.

FormData básico
formulario.addEventListener("submit", (evento) => {
  evento.preventDefault()

  const formData = new FormData(formulario)

  console.log(formData.get("usuario"))
  console.log(formData.get("correo"))
})
FormData y archivos

Si hay un <input type="file" name="foto">, formData.get("foto") devuelve el archivo

7️⃣ Enviar datos desde frontend​

Aunque no es objeto de evaluación ya que pertenece a la programación de BackEnd, es útil ver el patrón típico de envío con fetch para entender el flujo.

🟩 Envío como JSON (cuando no hay archivos)​

Enviar JSON con fetch (patrón típico)
formulario.addEventListener("submit", async (evento) => {
  evento.preventDefault()

  const datos = {
    usuario: formulario.elements["usuario"].value,
    correo: formulario.elements["correo"].value
  }

  // Ejemplo orientativo
  await fetch("/api/registro", {
    method: "POST",
    headers: { "Content-Type": "application/json" },
    body: JSON.stringify(datos)
  })
})

🟨 Envío como FormData (cuando hay archivos)​

Enviar FormData (patrón típico)
formulario.addEventListener("submit", async (evento) => {
  evento.preventDefault()

  const formData = new FormData(formulario)

  await fetch("/api/subida", {
    method: "POST",
    body: formData
  })
})
Importante

Con FormData no se suele poner Content-Type manualmente El navegador lo gestiona (incluyendo el boundary)

✅ Buenas prácticas con formularios​

Sigue estos consejos...
  1. Usa name en todos los campos si vas a usar form.elements o FormData
  2. Evita IDs duplicados y usa label con for por accesibilidad
  3. Intercepta con submit y preventDefault() si vas a validar o enviar con JS
  4. Valida con HTML primero y añade JS solo para reglas extra
  5. Limpia errores personalizados si usas setCustomValidity("")
  6. No confíes en la validación del cliente si los datos van a un servidor
