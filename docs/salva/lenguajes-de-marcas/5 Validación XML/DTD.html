5.1 DTD

DTD (Document Type Definition o Definición de Tipo de Documento) es un conjunto de reglas que especifica la estructura y el tipo de datos que puede contener un documento XML.Un DTD define los elementos y atributos permitidos, así como la relación y la estructura jerárquica entre ellos. Su función principal es proporcionar una descripción formal de la sintaxis y semántica de un documento XML, permitiendo que diferentes aplicaciones interpreten y validen los documentos XML de manera consistente.

Por lo tanto la validación de documentos XML mediante DTD consiste en emplear un conjunto de reglas que define qué elementos, atributos o entidades pueden aparecer en un documento y de qué manera. Un XML que cumpla todas las directrices indicadas en el DTD, se considerará “válido”.

1. Vinculación del DTD al XML​

Para que un documento XML se valide correctamente contra un DTD, es necesario establecer un vínculo entre ambos. Este vínculo indica que el XML va a “consultar” la definición que ofrece el DTD, comprobando si la cumple o no. Existen tres métodos principales para enlazar un DTD con un documento XML:

DTD interna​

En este caso, las reglas del DTD se incluyen directamente en el propio documento XML, justo después de la declaración <?xml ...?> y antes del elemento raíz. Para ello, se utiliza la etiqueta <!DOCTYPE ... [ ... ]>.

Ejemplo
Imagina que creamos un XML para describir videojuegos y queremos validarlo con un DTD interno:

<?xml version="1.0" encoding="UTF-8"?>
<!DOCTYPE videojuegos [
    <!ELEMENT videojuegos (videojuego+)>
    <!ELEMENT videojuego (titulo, genero, plataforma)>
    <!ELEMENT titulo (#PCDATA)>
    <!ELEMENT genero (#PCDATA)>
    <!ELEMENT plataforma (#PCDATA)>
]>
<videojuegos>
    <videojuego>
        <titulo>League of Legends</titulo>
        <genero>MOBA</genero>
        <plataforma>PC</plataforma>
    </videojuego>
    <videojuego>
        <titulo>God of War</titulo>
        <genero>Acción</genero>
        <plataforma>PlayStation</plataforma>
    </videojuego>
</videojuegos>
  1. Se declara el documento XML en la cabecera (<?xml version="1.0" encoding="UTF-8"?>).
  2. Después, dentro de <!DOCTYPE videojuegos [ ... ]> se especifican las reglas del DTD:
    • <!ELEMENT videojuegos (videojuego+)> significa que el elemento raíz videojuegos contendrá uno o varios (+) elementos videojuego.
    • <!ELEMENT videojuego (titulo, genero, plataforma)> establece que cada videojuego está compuesto exactamente por los subelementos titulo, genero y plataforma, en ese orden.
  3. Por último, se escribe el contenido real del XML, con la etiqueta raíz <videojuegos> que agrupa a varios <videojuego>.

En este caso, toda la definición necesaria para validar el XML está dentro del propio archivo.

fíjate

El nombre del DOCTYPE debe coincidir con el elemento raiz del documento XML

DTD externa (archivo aparte)​

Cuando se prevé que la misma definición del DTD se utilice para muchos documentos distintos (por ejemplo, varios archivos XML que describan playlists de música, o varias listas de videojuegos de diferentes épocas), lo más cómodo es ubicar el DTD en un archivo externo. Luego, desde el XML, simplemente hacemos referencia a ese archivo.

Supongamos que el archivo videojuegos.dtd contiene lo siguiente:

<!ELEMENT videojuegos (videojuego+)>
<!ELEMENT videojuego (titulo, genero, plataforma)>
<!ELEMENT titulo (#PCDATA)>
<!ELEMENT genero (#PCDATA)>
<!ELEMENT plataforma (#PCDATA)>

Para enlazarlo desde un documento XML, escribimos en la cabecera:

<?xml version="1.0" encoding="UTF-8"?>
<!DOCTYPE videojuegos SYSTEM "videojuegos.dtd">
<videojuegos>
    <videojuego>
        <titulo>Fortnite</titulo>
        <genero>Battle Royale</genero>
        <plataforma>PC</plataforma>
    </videojuego>
</videojuegos>
  • SYSTEM "videojuegos.dtd" indica que el DTD se encuentra en un archivo externo llamado videojuegos.dtd.
  • El procesador XML validará este documento en función de las reglas escritas en ese archivo.
Ventaja

Con un DTD externo, si necesitas modificar las reglas (añadir una nueva etiqueta, cambiar el orden de los elementos, etc.), no tienes que tocar todos los XML, sino solo el fichero .dtd.

DTD con identificador público (PUBLIC)​

Si el DTD está publicado en un lugar accesible (una URL o un sistema que defina identificadores públicos), se puede referenciar indicando un identificador público y, opcionalmente, la ubicación del DTD.

<!DOCTYPE videojuegos PUBLIC
    "-//MiOrganizacion//DTD Videojuegos 1.0//ES"
    "http://www.miorganizacion.com/dtd/videojuegos.dtd">
  • El primer parámetro ("-//MiOrganizacion//DTD Videojuegos 1.0//ES") es el identificador público.
  • El segundo ("http://www.miorganizacion.com/dtd/videojuegos.dtd") indica dónde se encuentra físicamente el DTD.

Este método se ve más a menudo en estándares o proyectos grandes, donde diferentes desarrolladores de todo el mundo van a utilizar el mismo DTD. En entornos educativos o pequeños, no es tan común, pero conviene conocerlo.

Comparativa de los métodos

  • DTD interna:

    • Todo está en un solo archivo XML.
    • Más rápido para casos sencillos o de aprendizaje.
    • No sirve bien para validar muchos documentos similares, pues habría que copiar y pegar el DTD dentro de cada uno.

  • DTD externa:

    • DTD independiente en un archivo .dtd.
    • Muy útil en proyectos donde se comparten las mismas reglas para numerosos XML.
    • Facilita el mantenimiento (modificas el DTD una sola vez).

  • DTD con identificador público:

    • Ideal para proyectos grandes o estándares.
    • Usado cuando el DTD se publica en un sitio remoto o un repositorio centralizado.

2. Tipos de Declaraciones​

Dentro de un DTD podemos encontrar cuatro tipos de declaraciones diferentes:

  • ELEMENT. Identifica elementos dentro de un documento XML.
  • ATTLIST. Identifica atributos dentro de un documento XML.
  • ENTITY. Identifica entidades declaradas dentro del DTD (similares a las constantes)
  • NOTATION. Define otro tipo de contenido, facilitando la inclusión de formatos binarios (imágenes, video, sonido…)

Por tanto, un documento XML será válido si, además de no tener errores de sintaxis, cumple lo indicado en las declaraciones de elementos, atributos, entidades y notaciones, de la DTD a la que esté asociado.

3. Ejemplo DTD simple​

Imagina un archivo tienda.dtd:

<!ELEMENT tienda (producto+)>
<!ELEMENT producto (nombre, precio)>
<!ELEMENT nombre (#PCDATA)>
<!ELEMENT precio (#PCDATA)>
<!ATTLIST producto
    categoria (videojuego|ropa|comida) "videojuego"
>

Y un XML que lo referencia de forma externa:

<?xml version="1.0" encoding="UTF-8"?>
<!DOCTYPE tienda SYSTEM "tienda.dtd">
<tienda>
    <producto categoria="ropa">
        <nombre>Camiseta de eSports</nombre>
        <precio>19.99</precio>
    </producto>
    <producto>
        <nombre>Super Smash Bros</nombre>
        <precio>59.99</precio>
    </producto>
</tienda>
  • Hemos colocado la lógica (elementos y atributos permitidos) en el archivo tienda.dtd.
  • En el XML, al incluir <!DOCTYPE tienda SYSTEM "tienda.dtd">, aseguramos que el procesador validará que:
    • El elemento raíz sea tienda.
    • Contenga uno o varios producto.
    • Cada producto tenga exactamente nombre y precio.
    • categoria sea opcional, pero si aparece, solo puede tomar los valores videojuego, ropa o comida.
    • Por defecto, si no se especifica categoria, es videojuego (según la definición del atributo).

De esta forma, cualquier documento que no cumpla esas reglas será rechazado como “no válido”. Por ejemplo, si en precio metes letras en lugar de números, el DTD no lo diferenciará de texto (ya que no define tipos numéricos), pero sí detectará si faltan etiquetas o si el orden de las mismas no coincide con lo definido.

En resumidas cuentas, vincular un DTD a un XML es tan sencillo como escribir la línea de <!DOCTYPE ...> con la forma que más se ajuste a tu proyecto.
