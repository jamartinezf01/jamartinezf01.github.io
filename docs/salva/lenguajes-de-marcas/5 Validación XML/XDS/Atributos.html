Atributos

En XSD (XML Schema Definition), los atributos son una parte fundamental para describir cómo se estructura y valida un documento XML. En pocas palabras, definen las “propiedades” de un elemento, permitiendo precisar qué valores puede (o debe) llevar un elemento como “metadatos” o “características” adjuntas.

1. Dónde se definen los atributos en XSD​

En un archivo XSD, los atributos se declaran dentro de un <xs:complexType> o directamente en un <xs:element> (si se quiere describir un tipo en línea). Por ejemplo:

<xs:element name="videojuego">
  <xs:complexType>
    <xs:sequence>
      <xs:element name="titulo" type="xs:string"/>
      <xs:element name="genero" type="xs:string"/>
    </xs:sequence>
    <xs:attribute name="id" type="xs:ID" use="required"/>
    <xs:attribute name="plataforma" type="xs:string" use="optional" default="PC"/>
  </xs:complexType>
</xs:element>

Aquí, videojuego define subelementos (<titulo>, <genero>) y atributos (id, plataforma).

2. Declaración básica de un atributo​

Para un atributo, se usa la etiqueta <xs:attribute> con propiedades clave:

  1. name: El nombre del atributo tal como aparecerá en el XML.
  2. type: El tipo de dato, por ejemplo, xs:string, xs:integer, etc.
  3. use: El modo de uso del atributo (ej. required, optional, o prohibited).
  4. default o fixed: Para asignar un valor por defecto o uno fijo.

Ejemplo sencillo:

<xs:attribute name="codigo" type="xs:string" use="required"/>

Significa que todo elemento que use este atributo “codigo” está obligado a especificarlo, y su valor será de tipo string.

3. Atributos con un xs:simpleType​

Si tenemos un tipo simple personalizado con restricciones (por ejemplo, una enumeración de valores), también podemos aplicarlo a un atributo. Por ejemplo:

<xs:simpleType name="tipoPlataforma">
  <xs:restriction base="xs:string">
    <xs:enumeration value="PC"/>
    <xs:enumeration value="Consola"/>
    <xs:enumeration value="Movil"/>
  </xs:restriction>
</xs:simpleType>

<xs:attribute name="plataforma" type="tipoPlataforma" use="optional"/>

De este modo, el atributo “plataforma” solo puede tomar los valores “PC”, “Consola” o “Movil”.

4. Atributos en un xs:complexType con xs:attribute​

Normalmente, en un tipo complejo (que admite subelementos), los atributos se definen fuera de <xs:sequence>, <xs:choice>, o <xs:all>, pero dentro del <xs:complexType>:

<xs:complexType name="personaType">
  <xs:sequence>
    <xs:element name="nombre" type="xs:string"/>
    <xs:element name="edad" type="xs:integer"/>
  </xs:sequence>
  <xs:attribute name="id" type="xs:ID" use="required"/>
  <xs:attribute name="genero" type="xs:string" use="optional" default="desconocido"/>
</xs:complexType>

El resultado permite un XML del estilo:

<persona id="P001" genero="Femenino">
  <nombre>Ana</nombre>
  <edad>30</edad>
</persona>

5. Modos de uso (use)​

En XSD, cuando defines un atributo dentro de un <xs:complexType> o directamente en un <xs:element>, el atributo use determina si ese atributo es obligatorio, opcional o queda prohibido para instancias específicas del tipo. A continuación se describen estos modos con ejemplos adicionales:

use="required"​

Esto indica que el atributo siempre debe aparecer en el XML, sin excepción.
Ejemplo:

<xs:element name="videojuego">
  <xs:complexType>
    <xs:sequence>
      <xs:element name="titulo" type="xs:string"/>
      <xs:element name="genero" type="xs:string"/>
    </xs:sequence>
    <xs:attribute name="id" type="xs:ID" use="required"/>
  </xs:complexType>
</xs:element>
info
  • Aquí, el atributo id es de tipo xs:ID y se marca con use="required".
  • Uso en el XML: 
    <videojuego id="VG001">
      <titulo>Celeste</titulo>
      <genero>Plataformas</genero>
    </videojuego>
    Si omites id="VG001", el documento sería inválido, pues id es obligatorio.

use="optional"​

Significa que el atributo puede o no aparecer en el elemento. Si no aparece, sigue siendo válido el documento.

Ejemplo:

<xs:element name="jugador">
  <xs:complexType>
    <xs:sequence>
      <xs:element name="nickname" type="xs:string"/>
      <xs:element name="ranking"  type="xs:integer" minOccurs="0"/>
    </xs:sequence>
    <xs:attribute name="avatar" type="xs:anyURI" use="optional"/>
  </xs:complexType>
</xs:element>
info
  • En el XML, el atributo avatar es opcional: 
    <jugador>
      <nickname>ProGamerX</nickname>
      <ranking>1</ranking>
    </jugador>

    <jugador avatar="http://miweb.com/avatars/progamer.png">
      <nickname>NoobMaster</nickname>
    </jugador>
    Ambos <jugador> son válidos; el primero no especifica avatar, el segundo sí.

use="prohibited"​

Indica que el atributo no debe aparecer. Esto suele usarse en casos donde estás redefiniendo o restringiendo un tipo en XSD y deseas eliminar un atributo previamente definido, haciendo que su aparición sea inválida.

Supongamos que tienes un tipo general productoType que define un atributo color. Luego quieres crear un tipo derivado que no admita ese atributo color. Ahí, use="prohibited" resulta útil.

Ejemplo:

<!-- Tipo base -->
<xs:complexType name="productoType">
  <xs:sequence>
    <xs:element name="nombre" type="xs:string"/>
  </xs:sequence>
  <xs:attribute name="color" type="xs:string" use="optional"/>
</xs:complexType>
  • Aquí, productoType admite (opcionalmente) un atributo color.

Ahora, se define un tipo derivado que no permita ese atributo:

<xs:complexType name="productoSinColor">
  <xs:complexContent>
    <xs:restriction base="productoType">
      <xs:sequence>
        <xs:element name="nombre" type="xs:string"/>
      </xs:sequence>
      <!-- Eliminamos el atributo color del tipo original -->
      <xs:attribute name="color" use="prohibited"/>
    </xs:restriction>
  </xs:complexContent>
</xs:complexType>
  • Aquí, color está prohibido. Si el XML de un <productoSinColor> intenta incluir color="rojo", será inválido.

Uso en el XML:

<productoSinColor>
  <nombre>Tarjeta de crédito</nombre>
</productoSinColor>

Correcto.

<productoSinColor color="rojo">
  <nombre>Tarjeta de crédito</nombre>
</productoSinColor>

Inválido, pues se prohibió el atributo color.

6. Valores por defecto y fijos​

  • default: Si el atributo no se especifica en el XML, el procesador asume un valor predeterminado. Por ejemplo:

    <xs:attribute name="color" type="xs:string" use="optional" default="rojo"/>

    Si “color” no aparece, se interpreta como “rojo”.

  • fixed: Obliga a que el atributo tenga siempre el mismo valor; si se pone otro, el documento será inválido.

    <xs:attribute name="version" type="xs:string" use="optional" fixed="1.0"/>

7. Tipos de atributo en XSD​

Al igual que en elementos, los atributos pueden usar los tipos predefinidos de XSD (por ejemplo, xs:string, xs:boolean, xs:decimal, xs:date, etc.) o tipos personalizados basados en <xs:simpleType>. También se puede utilizar xs:ID, xs:IDREF, xs:IDREFS en atributos, tal como sucede en DTD, pero con la ventaja de que se pueden imponer restricciones adicionales (patrones, enumeraciones, etc.).

Ejemplo:

<xs:attribute name="codigoProducto" type="xs:ID" use="required"/>
<xs:attribute name="refRelacionado" type="xs:IDREF"/>

Esto crea la relación de identificadores en XSD de forma parecida a DTD, pero dentro del contexto de un esquema más robusto.

8. Atributos Globales vs. Locales​

Lo más habitual es definir los atributos localmente dentro de <xs:complexType> o directamente en <xs:element>. Sin embargo, XSD también permite declarar atributos a nivel global (igual que se hace con elementos globales), de manera que puedan referenciarse en distintos tipos:

<xs:attribute name="idDocumento" type="xs:ID"/>

<xs:complexType name="documentoType">
  <xs:sequence> ... </xs:sequence>
  <xs:attribute ref="idDocumento" use="required"/>
</xs:complexType>
info

Aquí, el atributo idDocumento está declarado globalmente (fuera de cualquier <complexType>) y luego referenciado con ref="idDocumento".

9. Ejemplo Completo​

<xs:element name="orden">
  <xs:complexType>
    <xs:sequence>
      <xs:element name="articulo" minOccurs="1" maxOccurs="unbounded">
        <xs:complexType>
          <xs:attribute name="codigo" type="xs:string" use="required"/>
          <xs:attribute name="cantidad" type="xs:integer" use="required" default="1"/>
        </xs:complexType>
      </xs:element>
    </xs:sequence>
    <xs:attribute name="numOrden" type="xs:ID" use="required"/>
    <xs:attribute name="estado" type="xs:string" use="optional" fixed="pendiente"/>
  </xs:complexType>
</xs:element>
info
  • <orden>:
    • Atributos de orden: numOrden (ID obligatorio) y estado (siempre “pendiente”).
    • Subelementos <articulo> con 2 atributos (codigo, cantidad). Si cantidad no se especifica, se asume “1”.

Así un XML podría ser:

<orden numOrden="ORD001" estado="pendiente">
  <articulo codigo="A1001" cantidad="2"/>
  <articulo codigo="B2003"/>
</orden>

En el segundo <articulo>, cantidad se interpreta como “1” por la definición default="1".
