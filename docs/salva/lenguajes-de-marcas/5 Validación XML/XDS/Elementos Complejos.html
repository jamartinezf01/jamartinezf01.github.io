Elementos Complejos

Los elementos complejos (o compuestos) en XSD (XML Schema Definition) son aquellos que pueden contener subelementos y/o atributos, a diferencia de los elementos simples, que solo admiten texto. Para muchos escenarios actuales (como describir tu lista de videojuegos, tus colecciones de música o tus streamers favoritos), este concepto es esencial para organizar datos complejos y jerárquicos.

1. ¿Qué es un elemento complejo?​

En un esquema XSD, un elemento complejo es aquel que:

  1. Puede contener uno o más subelementos (por ejemplo, un conjunto de etiquetas en su interior).
  2. Puede tener atributos opcionales u obligatorios.
  3. Puede especificar la cardinalidad (cuántas veces pueden aparecer ciertos subelementos), el orden (mediante secuencias o elecciones), y el tipo de datos.

El elemento complejo se define usando <xs:complexType>. Existen varias formas de estructurarlo, dependiendo de si su contenido es:

  • Completamente vacío (EMPTY).
  • Solo texto (y atributos), llamado contenido “simple”.
  • Solo subelementos (y atributos).
  • Mixto (texto + subelementos).

2. Declaración de un elemento complejo​

Típicamente, se escribe:

<xs:element name="playlist">
  <xs:complexType>
    <xs:sequence>
      <xs:element name="cancion" type="xs:string" maxOccurs="unbounded"/>
    </xs:sequence>
  </xs:complexType>
</xs:element>

En este ejemplo, <playlist> es un elemento complejo porque contiene subelementos (cancion).

3. Elementos Complejos vacíos​

Un elemento complejo vacío es un elemento que no puede contener ni texto ni elementos hijos. Su estructura es compleja porque se define con <xs:complexType>, pero está restringido a ser vacío. Se utilizan en situaciones donde se desea definir elementos con atributos, pero sin contenido dentro de ellos.

Ejemplo:

  • XSD
  • XML
<xs:element name="usuario">
  <xs:complexType>
    <xs:attribute name="id" type="xs:string" use="required"/>
    <xs:attribute name="rol" type="xs:string" use="optional"/>
  </xs:complexType>
</xs:element>

      <usuario id="12345" rol="admin"/>

El elemento usuario no tiene contenido ni elementos hijos pero posee dos atributos (id obligatorio y rol opcional).

4. Elementos Complejos con Contenido Simple​

Un elemento complejo con contenido simple es aquel que tiene contenido y atributos pero sin subelementos (no permite elementos hijos). Se utiliza cuando un dato principal debe ir acompañado de metadatos o información adicional en forma de atributos.

  • XSD
  • XML
<xs:element name="precio">
  <xs:complexType>
    <xs:simpleContent>
      <xs:extension base="xs:decimal">
        <xs:attribute name="moneda" type="xs:string" use="required"/>
      </xs:extension>
    </xs:simpleContent>
  </xs:complexType>
</xs:element>
  <precio moneda="€">19.99</precio>

Precio contiene un valor decimal como contenido principal. Incluye el atributo moneda que es obligatorio (use="required"). No permite elementos hijos, solo texto y atributos.

5. Tipos de Elementos Complejos con subelementos​

Elementos complejos con solo subelementos​

Cuando se habla de elementos complejos con solo subelementos, se hace referencia a aquellos que no contienen texto libre directamente, ni atributos, sino únicamente hijos (otros elementos). Suelen aparecer en estructuras fuertemente jerárquicas, como un <pedido> que contiene varios <producto>, o un <torneo> que contiene <equipo> y <jugador>.

En XSD, para describir la presencia de únicamente subelementos (sin texto ni atributos), lo habitual es:

  1. Utilizar un <xs:complexType>.
  2. Dentro de él, anidar uno de los indicadores de orden principales: <xs:sequence>, <xs:choice> o <xs:all>.
  3. Declarar cada subelemento con su tipo y, si hace falta, sus reglas de multiplicidad (minOccurs, maxOccurs).
<xs:complexType>
  <xs:sequence>
    <xs:element name="titulo"  type="xs:string"/>
    <xs:element name="autor"   type="xs:string"/>
  </xs:sequence>
</xs:complexType>
  • <xs:sequence>: exige que los subelementos aparezcan en un orden concreto.
  • <xs:choice>: deja que aparezca uno u otro subelemento de una lista.
  • <xs:all>: todos los subelementos pueden aparecer en cualquier orden, aunque con ciertas restricciones (p. ej., maxOccurs=1).

A continuación se detallan los componentes relevantes...

A) Uso de <xs:sequence>​

La etiqueta <xs:sequence> especifica que sus hijos deben aparecer en un orden exacto dentro del elemento. Siendo un elemento complejo con solo subelementos, no caben atributos ni texto directo, de modo que todo el contenido se distribuye en la secuencia.

Ejemplo:

  • XSD
  • XML
<xs:element name="concierto">
  <xs:complexType>
    <xs:sequence>
      <xs:element name="artista" type="xs:string"/>
      <xs:element name="fecha"   type="xs:date"/>
      <xs:element name="lugar"   type="xs:string"/>
    </xs:sequence>
  </xs:complexType>
</xs:element>
<concierto>
<artista>Taylor Swift</artista>
<fecha>2024-07-18</fecha>
<lugar>Estadio Central</lugar>
</concierto>

En este caso, <concierto> es un elemento complejo con solo los subelementos <artista>, <fecha> y <lugar>, y deben aparecer en ese orden exacto: primero <artista>, luego <fecha>, luego <lugar>. Si en el XML intentas poner <lugar> antes de <fecha>, la validación fallará.

B) Uso de <xs:choice>​

Con <xs:choice>, indicas que solo uno de los hijos listados puede aparecer, o si le añades cardinalidad (maxOccurs > 1), varios, pero siempre solo uno de la lista cada vez.

Ejemplo:

  • XSD
  • XML
<xs:element name="mensaje">
  <xs:complexType>
    <xs:choice maxOccurs="unbounded">
      <xs:element name="sms" type="xs:string"/>
      <xs:element name="email" type="xs:string"/>
    </xs:choice>
  </xs:complexType>
</xs:element>
<mensaje>
<sms>¡Hola, ya llegué!</sms>
<email>Te escribo para confirmar la cita.</email>
<sms>Este es el recordatorio final.</sms>
</mensaje>

En este ejemplo, <mensaje> puede contener una o varias (debido a maxOccurs="unbounded") subetiquetas que sean <sms> o <email>, en cualquier orden. Sigues sin tener texto libre ni atributos, únicamente subelementos. En el XML , cada subelemento puede ser sms o email. No existe un atributo ni texto puro dentro de <mensaje>, solo esas etiquetas.

C) Uso de <xs:all>​

<xs:all> indica que sus subelementos pueden aparecer en cualquier orden, y cada uno se admite al máximo una vez por defecto (o con minOccurs definido para 0 o 1, pero no más). Es la forma más estricta de “contenidos paralelos”: todos sus hijos pueden ocurrir, pero sin multiplicidades grandes.

Ejemplo:

<xs:element name="streamerProfile">
  <xs:complexType>
    <xs:all>
      <xs:element name="nickname" type="xs:string" minOccurs="1"/>
      <xs:element name="seguidorDestacado" type="xs:string" minOccurs="0"/>
      <xs:element name="canalEnlace" type="xs:anyURI" minOccurs="1"/>
    </xs:all>
  </xs:complexType>
</xs:element>

Esto significa que dentro de <streamerProfile>, pueden aparecer <nickname>, <seguidorDestacado> (opcional) y <canalEnlace>, pero en cualquier orden, eso sí, cada uno como máximo una vez.

Elementos complejos con atributos y subelementos​

Puedes añadir atributos dentro del <xs:complexType> pero fuera de <xs:sequence> (o <xs:choice>, <xs:all>). Por ejemplo, en un XML donde describes a tu “streamer” favorito:

<xs:element name="streamer">
  <xs:complexType>
    <xs:sequence>
      <xs:element name="nickname" type="xs:string"/>
      <xs:element name="plataforma" type="xs:string"/>
    </xs:sequence>
    <xs:attribute name="suscriptores" type="xs:integer" use="optional"/>
  </xs:complexType>
</xs:element>

6. Contenido mixto (texto + subelementos)​

A veces quieres un elemento que contenga texto pero también subelementos entremezclados. Imagínate un post de un blog, donde parte es texto y parte son elementos (imágenes, menciones, etc.). Esto se llama contenido mixto y se declara con mixed="true":

<xs:element name="post">
  <xs:complexType mixed="true">
    <xs:sequence>
      <xs:element name="imagen" minOccurs="0" maxOccurs="unbounded"/>
      <xs:element name="enlace" minOccurs="0" maxOccurs="unbounded"/>
    </xs:sequence>
    <xs:attribute name="autor" type="xs:string"/>
  </xs:complexType>
</xs:element>

Con mixed="true", el texto y los subelementos (<imagen>, <enlace>) pueden intercalarse en cualquier orden en tu <post>.

7. Ejemplo “Ficha de Videojuego”​

Imagina un XSD para describir los juegos que tienes en tu lista de “pendientes”:

  • XSD
  • XML
<xs:element name="pendientes">
  <xs:complexType>
    <xs:sequence>
      <xs:element name="juego" maxOccurs="unbounded">
        <xs:complexType>
          <xs:sequence>
            <xs:element name="titulo"     type="xs:string"/>
            <xs:element name="desarrolladora" type="xs:string"/>
            <xs:element name="plataforma" type="xs:string"/>
            <xs:element name="fecha_lanzamiento" type="xs:date" minOccurs="0"/>
          </xs:sequence>
          <xs:attribute name="completado" type="xs:boolean" use="optional" default="false"/>
        </xs:complexType>
      </xs:element>
    </xs:sequence>
    <xs:attribute name="propietario" type="xs:string" use="required"/>
  </xs:complexType>
</xs:element>
<pendientes propietario="MiNombre">
<juego completado="true">
  <titulo>The Legend of Zelda: Breath of the Wild</titulo>
  <desarrolladora>Nintendo</desarrolladora>
  <plataforma>Nintendo Switch</plataforma>
  <fecha_lanzamiento>2017-03-03</fecha_lanzamiento>
</juego>
<juego>
  <titulo>Hollow Knight</titulo>
  <desarrolladora>Team Cherry</desarrolladora>
  <plataforma>PC</plataforma>
</juego>
</pendientes>
info
  • <pendientes> es un elemento complejo, con subelementos <juego> y un atributo propietario.
  • <juego> es a su vez otro elemento complejo, con subelementos <titulo>, <desarrolladora>, <plataforma>, <fecha_lanzamiento>, y un atributo completado.
  • <fecha_lanzamiento> se ha hecho opcional (minOccurs="0"), por si no conoces la fecha exacta.

8. Hacer referencia a tipos complejos​

También puedes definir un <xs:complexType> global y luego referenciarlo desde varios <xs:element>. Esto se usa mucho cuando el mismo patrón de subelementos/atributos se repite. Por ejemplo:

<xs:complexType name="infoStreamerType">
  <xs:sequence>
    <xs:element name="nickname" type="xs:string"/>
    <xs:element name="plataforma" type="xs:string"/>
  </xs:sequence>
  <xs:attribute name="suscriptores" type="xs:integer"/>
</xs:complexType>

<xs:element name="streamer" type="infoStreamerType"/>
<xs:element name="colaborador" type="infoStreamerType"/>

Aquí, tanto <streamer> como <colaborador> comparten la misma estructura infoStreamerType.

9. Buenas Prácticas y Consejos​

emplea estos consejos...
  1. Aprovecha xs:sequence, xs:choice, y xs:all para describir el orden y la combinación de subelementos.
  2. Use minOccurs y maxOccurs: Controla cuántas veces puede aparecer un subelemento (0, 1, ilimitado).
  3. Combina elementos complejos con atributos para describir distintos niveles de detalle.
  4. Mantenlo organizado: Si un patrón se repite, define un complexType global para reutilizarlo.
  5. Documenta tus elementos y atributos con anotaciones (<xs:annotation><xs:documentation>) para explicar a otros (y a ti mismo) por qué se definió esa estructura.
