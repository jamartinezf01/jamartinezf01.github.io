Reglas de Seguridad

Las Reglas de Seguridad de Firebase Realtime Database te permiten controlar de forma declarativa quién puede leer, escribir o realizar operaciones más avanzadas en tu base de datos. Estas reglas se basan en la información de autenticación del usuario (auth) y en la estructura de los datos, lo que te facilita restringir el acceso de acuerdo a la lógica de tu aplicación. Con un buen dominio de las reglas, te asegurarás de que tu aplicación sea segura y de que cada usuario solo acceda a la información a la que deba tener derecho.

1. Dónde se Definen las Reglas​

En la Consola de Firebase, navega a Build → Realtime Database y busca la pestaña Rules. Allí encontrarás un editor donde podrás escribir tus reglas en formato JSON.

{
  "rules": {
    ".read": "condition",
    ".write": "condition"
  }
}

También puedes definir reglas más específicas para cada nodo de la base de datos. Por ejemplo:

{
  "rules": {
    "anuncios": {
      ".read": "auth != null",
      ".write": "auth != null"
    },
    "usuarios": {
      ".read": "auth != null",
      ".write": "auth.uid === $uid"
    }
  }
}
En este ejemplo:
  • Para la rama anuncios, solo se permite la lectura y escritura a usuarios autenticados (auth != null).
  • Para la rama usuarios, solo se permite la lectura a usuarios autenticados, y la escritura únicamente se concede si el uid del usuario autenticado coincide con el nodo en la base de datos ($uid se refiere al nombre del nodo). Este tipo de regla se usa, por ejemplo, para que cada usuario solo pueda escribir en su propio perfil.

2. Cómo Funcionan las Condiciones​

En cada propiedad (.read, .write, etc.), puedes usar expresiones booleanas que devuelven true o false. Entre las variables y funciones más habituales están:

  • auth: Objeto que representa la información de autenticación del usuario (cuando el usuario está autenticado). Si el usuario no está autenticado, auth será null.

  • auth.uid: ID único del usuario que inicia sesión con Firebase Authentication.

  • $variable: Parámetros en la ruta que permiten reglas dinámicas.
    Por ejemplo, si tu regla está en usuarios/$uid, la variable $uid tomará el valor de cada nodo hijo en usuarios.

  • Operadores lógicos: && (AND), || (OR), ! (NOT), entre otros.

  • Otras funciones: Por ejemplo, root.child('path').val(), now, newData, data, etc., para reglas avanzadas.

info

Para saber más sobre las reglas de seguridad visita la Documentación de Firebase

3. Ejemplos de Reglas Comunes​

Solo usuarios autenticados pueden leer y escribir en toda la base de datos:​

{
  "rules": {
    ".read": "auth != null",
    ".write": "auth != null"
  }
}
Explicación
  • auth != null significa que se requiere un usuario autenticado.

Leer datos de forma pública, pero solo usuarios autenticados pueden escribir:​

{
  "rules": {
    ".read": true,
    ".write": "auth != null"
  }
}
Explicación
  • ".read": true significa acceso de lectura abierto a todos.
  • ".write": "auth != null" restringe las escrituras a usuarios registrados.

Reglas específicas para diferentes nodos:​

{
  "rules": {
    "perfiles": {
      "$uid": {
        // Permite solo al usuario dueño de este perfil leer/escribir
        ".read": "auth != null && auth.uid === $uid",
        ".write": "auth != null && auth.uid === $uid"
      }
    },
    "mensajesPublicos": {
      // Lectura abierta a todos, escritura solo a usuarios registrados
      ".read": true,
      ".write": "auth != null"
    }
  }
}
info
  • Cada rama (perfiles, mensajesPublicos) tiene reglas propias.
  • En perfiles/$uid, se verifica que auth.uid sea igual al $uid del nodo para dar acceso.

4. Validación de Datos​

Además de permitir o denegar acceso, las reglas también pueden validar la estructura y el tipo de los datos que se escriben. Por ejemplo, si quieres asegurar que en anuncios cada nodo tenga la propiedad titulo de tipo cadena y un mensaje también de tipo cadena, puedes usar:

{
  "rules": {
    "anuncios": {
      "$anuncioId": {
        ".write": "auth != null",
        "titulo": {
          ".validate": "newData.isString() && newData.val().length > 0"
        },
        "mensaje": {
          ".validate": "newData.isString() && newData.val().length > 0"
        }
      }
    }
  }
}
En este ejemplo:
  • ".validate" verifica la estructura y el contenido del nodo:
    • newData.isString() obliga a que los valores sean cadenas.
    • newData.val().length > 0 obliga a que no estén vacías.
  • Si la validación falla, la escritura se rechaza automáticamente.

5. Prueba de Reglas​

Para asegurarte de que tus reglas están configuradas correctamente, utiliza la pestaña “Rules” → “Test Rules” en la consola de Firebase. Allí puedes simular lecturas/escrituras (pasando un uid o sin él) para ver si se permitiría o denegaría el acceso.

También puedes usar la Firebase CLI con el comando firebase emulators:start --only database y luego escribir reglas en tiempo real, probar lecturas/escrituras con requests locales o usar el Emulator UI para verificar el comportamiento de las reglas sin riesgo de alterar la base de datos en producción.

Buenas Prácticas​

sigue estas reglas
  1. Principio de Privilegios Mínimos: Permite solo lo necesario. Evita reglas demasiado abiertas como ".read": true, ".write": true.
  2. Separar Ramas Sensibles: Si tienes datos que requieren protección adicional (p.ej. datos personales, contraseñas cifradas, etc.), ponlos en una rama con reglas más estrictas.
  3. Validación: Además de restringir acceso, aprovecha las reglas para validar el formato de los datos. Esto reduce errores y abusos.
  4. Autenticación: Combinar auth != null con validación de auth.uid es clave para asociar datos a usuarios concretos.
  5. Testing y Emuladores: Antes de desplegar reglas a producción, pruébalas localmente con el emulador o la sección Test Rules de la consola.

Diagrama​

  1. Una solicitud de lectura o escritura llega a la base de datos.
  2. Se buscan reglas específicas para el nodo en cuestión. Si no las hay, se buscan en el ancestro inmediato (subiendo hasta la raíz).
  3. La condición (por ejemplo auth != null && auth.uid === $uid) se evalúa.
  4. Si evalúa a true, se permite la operación; de lo contrario, se deniega.
