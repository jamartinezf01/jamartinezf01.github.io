Consultas

Aunque Firebase Realtime Database es una base de datos NoSQL basada en un árbol JSON, ofrece cierta funcionalidad de consultas para filtrar, ordenar y limitar los datos que obtenemos. A continuación, veremos las principales formas de hacer consultas usando el SDK de Android (Java).

Las consultas en Firebase Realtime Database se basan en ordenar los datos por un campo (orderByChild) y filtrar con métodos de rango (startAt, endAt, equalTo), además de limitar la cantidad de resultados (limitToFirst, limitToLast). Aunque no ofrecen la complejidad de un SQL o de Firestore (sin múltiples filtros o consultas muy avanzadas), sí permiten cubrir muchos casos comunes: listas ordenadas, búsqueda por prefijo de cadenas o rango de valores, y paginaciones sencillas.

1. Concepto de Query​

En Realtime Database, las consultas no se realizan mediante un lenguaje como SQL, sino empleando métodos encadenados sobre una referencia (DatabaseReference) para ordenar y filtrar. Estos métodos devuelven un objeto Query.

Ejemplo de patrón:

Query consulta = mDatabase.child("anuncios")
    .orderByChild("titulo")
    .startAt("A")    // Empieza donde el campo "titulo" >= "A"
    .endAt("M")      // Termina donde el campo "titulo" <= "M"
    .limitToFirst(10);

A partir de esa consulta, podemos emplear los métodos de lectura (addListenerForSingleValueEvent, addValueEventListener, etc.) para obtener resultados filtrados.

2. Métodos de Ordenación​

  • orderByChild(String path)

    • Ordena los resultados en función de un campo hijo específico. Por ejemplo, orderByChild("precio") ordena los nodos según el valor del campo precio.

  • orderByKey()

    • Ordena los resultados según la clave (key) de cada nodo.

  • orderByValue()

    • Ordena los resultados según el valor del nodo en caso de que el nodo sea un valor simple (p.ej., si guardaste un entero o una cadena directamente).

  • orderByPriority()

    • Ordena los resultados según la prioridad del nodo. (Menos usado; la prioridad es una funcionalidad previa de Realtime Database).
Importante

Solo una ordenación se puede aplicar por consulta.
No existen las consultas complejas “multiorden” o “and/or” como en SQL o Firestore.

3. Métodos de Filtro (Rango)​

Una vez que eliges un método de orden (orderByChild, etc.), puedes acotar los resultados por rango:

  • startAt(value)

    • Devuelve solo los nodos cuyo valor (o el campo usado en orderByChild) es mayor o igual a value.

  • endAt(value)

    • Devuelve solo los nodos cuyo valor es menor o igual a value.

  • equalTo(value)

    • Devuelve solo los nodos cuyo valor es igual a value.

Se suelen combinar con orderByChild(...) para filtrar según un campo en particular.

Ejemplo:

// Queremos los anuncios en los que "precio" sea >= 10 y <= 50
Query anunciosConPrecio = mDatabase.child("anuncios")
    .orderByChild("precio")
    .startAt(10)
    .endAt(50);

4. Métodos de Limitación (Paginación Sencilla)​

Para obtener solo una cantidad determinada de resultados, se usan:

  • limitToFirst(int limit)

    • Devuelve los primeros limit elementos de la lista ordenada.

  • limitToLast(int limit)

    • Devuelve los últimos limit elementos.

Ejemplo:

// Pedir los 5 primeros anuncios (según orden actual)
Query primerosCinco = mDatabase.child("anuncios")
    .orderByChild("fecha")
    .limitToFirst(5);

5. Ejemplos Prácticos de Consultas​

Ordenar por un Campo y Limitar el Número de Resultados​

DatabaseReference ref = FirebaseDatabase.getInstance().getReference().child("productos");

// Ordena productos por precio y devuelve solo los primeros 10
Query consulta = ref.orderByChild("precio").limitToFirst(10);

// Luego, escuchamos la consulta
consulta.addListenerForSingleValueEvent(new ValueEventListener() {
    @Override
    public void onDataChange(@NonNull DataSnapshot snapshot) {
        for (DataSnapshot ds : snapshot.getChildren()) {
            Producto producto = ds.getValue(Producto.class);
            if (producto != null) {
                Log.d("Consulta", "Producto: " + producto.getNombre() 
                                  + ", Precio: " + producto.getPrecio());
            }
        }
    }

    @Override
    public void onCancelled(@NonNull DatabaseError error) {
        Log.e("Consulta", "Error en la consulta", error.toException());
    }
});

Filtrar por un Rango de Valores​

// Queremos los "anuncios" donde el campo "titulo" empiece por "A" hasta "C"
Query anunciosA_C = mDatabase.child("anuncios")
    .orderByChild("titulo")
    .startAt("A")
    .endAt("C\uf8ff"); 
// "\uf8ff" se usa a veces para lograr que incluya cadenas que comienzan con "C"

anunciosA_C.addListenerForSingleValueEvent(new ValueEventListener() {
    @Override
    public void onDataChange(@NonNull DataSnapshot dataSnapshot) {
        // Recorremos los resultados
    }

    @Override
    public void onCancelled(@NonNull DatabaseError databaseError) {
    }
});

Suele usarse el truco "\uf8ff" para abarcar todas las cadenas que inician con un patrón.

Buscar por Valor Exacto​

Si quieres obtener todos los nodos que tengan un campo tipo con valor "oferta":

Query soloOfertas = mDatabase.child("productos")
    .orderByChild("tipo")
    .equalTo("oferta");

soloOfertas.addListenerForSingleValueEvent(new ValueEventListener() {
    ...
});

6. Combinar Consultas con Listeners en Tiempo Real​

Las consultas también pueden usarse con addValueEventListener o addChildEventListener para recibir actualizaciones dinámicas. Por ejemplo:

Query consultaEnTiempoReal = mDatabase.child("mensajes")
    .orderByChild("timestamp")
    .limitToLast(20);

consultaEnTiempoReal.addValueEventListener(new ValueEventListener() {
    @Override
    public void onDataChange(@NonNull DataSnapshot snapshot) {
        // Recibir los 20 últimos mensajes, y se actualizará 
        // automáticamente a medida que lleguen nuevos
    }

    @Override
    public void onCancelled(@NonNull DatabaseError error) {}
});

7. Limitaciones de las Consultas​

consultas en realtime database
  1. Una sola ordenación por consulta.
  2. Filtros sencillos (startAt, endAt, equalTo). No hay operadores como AND, OR, múltiples condiciones ni búsquedas complejas.
  3. Indices: Para grandes conjuntos de datos, puede ser necesario crear índices en las reglas de seguridad para mejorar el rendimiento.
  4. Funcionalidad: Menos potente que Firestore a nivel de queries. Si necesitas consultas muy complejas, tal vez Firestore se ajuste mejor.
