Estructura de Datos

En Cloud Firestore, los datos se organizan bajo un modelo NoSQL orientado a documentos y colecciones. Este enfoque ofrece gran flexibilidad, permitiendo adaptar la estructura a las necesidades de la aplicación sin necesidad de migraciones complejas. A continuación, se explica en detalle cómo se gestionan las colecciones, documentos y subcolecciones, así como los tipos de campos que puede manejar Firestore.

1. Conceptos fundamentales​

Firestore emplea dos unidades principales: colecciones y documentos.

  1. Colección

    • Es un contenedor de documentos.
    • Cada colección se identifica por un nombre único dentro de la base de datos (por ejemplo, usuarios, notas, productos).
    • No puede contener directamente otros documentos sin antes identificarlos (es decir, no se almacenan “documentos sueltos” fuera de colecciones).

  2. Documento

    • Representa una entidad de datos (por ejemplo, un usuario, una nota, un pedido).
    • Tiene un identificador único (ID) asignado por Firestore o especificado por el desarrollador.
    • Almacena campos con valores (por ejemplo, nombre, edad, fechaCreacion, listaTareas, etc.).
    • Puede contener subcolecciones, que a su vez pueden contener más documentos, y así recursivamente.

  3. Subcolección

    • Una subcolección es una colección anidada dentro de un documento.
    • Sirve para modelar relaciones jerárquicas de manera ordenada y escalable.
    • Ejemplo: Si tienes una colección usuarios con un documento userId123, dentro de ese documento podrías tener una subcolección pedidos.

  4. Campos

    • Son los atributos guardados en un documento.
    • Pueden ser de distintos tipos: String, Number, Boolean, Date, GeoPoint, Array, Map, etc.

2. Creación y administración de colecciones​

Crear o referenciar una colección​

En Java (Android), para obtener una referencia a una colección se usa:

FirebaseFirestore db = FirebaseFirestore.getInstance();
CollectionReference usuariosRef = db.collection("usuarios");
info

Si la colección usuarios no existe aún, Firestore la creará implícitamente al momento de escribir el primer documento. No es necesario “crear” las colecciones de manera explícita.

Además, existe la posibilida de usar la consola Firebase para Añadir una colección:

  • En la sección Datos de Firestore pulsa sobre Iniciar Colección

  • Despues, agrega el primer documento.

  • Podrás ver en la consola las colecciones con sus documentos relacionados...

Añadir un documento a una colección​

Como hemos visto anteriormente, existe la posibilida de añadir docuentos a las colecciones mediante la consola Firebase.

Aunque también podemos relizarlo mediante código...

Map<String, Object> nuevoUsuario = new HashMap<>();
nuevoUsuario.put("nombre", "Carlos");
nuevoUsuario.put("edad", 30);

usuariosRef.add(nuevoUsuario)
    .addOnSuccessListener(documentReference -> {
        Log.d("Firestore", "Documento creado con ID: " + documentReference.getId());
    })
    .addOnFailureListener(e -> {
        Log.e("Firestore", "Error al crear documento", e);
    });
explicación
  • add(...) crea el documento con un ID automático asignado por Firestore.
  • Si deseas asignar un ID personalizado, puedes usar document("miIdPersonalizado") y luego set(...).

3. Documentos y sus campos​

Los documentos representan el núcleo de la información en Firestore. Cada documento:

  • Puede tener un número ilimitado de campos.
  • Puede anidar estructuras de tipo Map y Array para representar datos más complejos.
  • Tiene un tamaño máximo recomendado de 1 MiB (incluyendo campos y metadatos).

Crear un documento con ID personalizado​

Map<String, Object> producto = new HashMap<>();
producto.put("nombre", "Camiseta");
producto.put("precio", 19.99);

db.collection("productos")
  .document("camiseta123")
  .set(producto)
  .addOnSuccessListener(aVoid -> {
      Log.d("Firestore", "Documento creado/actualizado con ID: camiseta123");
  })
  .addOnFailureListener(e -> {
      Log.e("Firestore", "Error al escribir documento", e);
  });
info

Si el documento no existía, se crea. Si ya existía, se sobrescribe (actualiza sus campos a los valores indicados).

Tipos de datos soportados​

Firestore permite varios tipos de datos comunes:

  1. String: Texto Unicode.
  2. Boolean: true o false.
  3. Number: Puede ser long (entero de 64 bits) o double (coma flotante).
  4. Timestamp (Date): Almacena fecha y hora, compatible con java.util.Date.
  5. Array: Lista ordenada de valores (puede combinar diferentes tipos).
  6. Map: Estructura anidada de pares clave-valor.
  7. GeoPoint: Representa coordenadas geográficas (latitud, longitud).
  8. Reference: Apunta a otro documento en la misma base de datos.
  9. Null: Valor nulo.

4. Subcolecciones: organización jerárquica​

Para modelar estructuras complejas o relaciones de uno-a-muchos (por ejemplo, los pedidos de un usuario), Firestore permite subcolecciones dentro de un documento.

Ejemplo: Usuarios y Pedidos​

FirebaseFirestore db = FirebaseFirestore.getInstance();

// Referencia a la colección usuarios
CollectionReference usuariosRef = db.collection("usuarios");

// Documento específico de un usuario
DocumentReference usuarioDoc = usuariosRef.document("userId123");

// Subcolección de pedidos dentro del documento userId123
CollectionReference pedidosRef = usuarioDoc.collection("pedidos");

// Agregar un pedido a la subcolección
Map<String, Object> nuevoPedido = new HashMap<>();
nuevoPedido.put("fecha", new Date());
nuevoPedido.put("total", 59.99);

pedidosRef.add(nuevoPedido)
    .addOnSuccessListener(documentReference -> {
        Log.d("Firestore", "Pedido creado con ID: " + documentReference.getId());
    })
    .addOnFailureListener(e -> {
        Log.e("Firestore", "Error al crear pedido", e);
    });
Ventajas de las subcolecciones:
  • Mantienes una relación “jerárquica” clara: cada usuario tiene su propia colección de pedidos.
  • Puedes escalar cada subcolección de forma independiente.
  • Reglas de seguridad más simples: Podrías restringir que cada usuario solo acceda a su subcolección de pedidos.

Acceder a documentos de subcolección​

Para leer un documento en la subcolección:

pedidosRef.document("pedidoId456")
    .get()
    .addOnSuccessListener(documentSnapshot -> {
        if (documentSnapshot.exists()) {
            Log.d("Firestore", "Pedido: " + documentSnapshot.getData());
        } else {
            Log.d("Firestore", "Documento no existe");
        }
    })
    .addOnFailureListener(e -> {
        Log.e("Firestore", "Error al leer pedido", e);
    });

5. Profundidad y diseño de la estructura​

Aunque Firestore permite anidar subcolecciones de forma ilimitada, es recomendable no abusar de la profundidad. Un diseño con demasiados niveles puede complicar las consultas y la seguridad. En general, busca un equilibrio:

  • Usa subcolecciones cuando necesites agrupar datos muy relacionados y potencialmente extensos (por ejemplo, múltiples documentos hijos).
  • Usa campos tipo Map si la anidación es relativamente pequeña y no necesitarás consultas específicas en esos campos.
  • Ten en cuenta la desnormalización: en bases de datos NoSQL, a veces es preferible duplicar datos para facilitar lecturas rápidas en lugar de un modelo relacional puro.

6. Ejemplo: Estructura Usuarios → Subcolección Pedidos​

public class FirestoreExample {

    private FirebaseFirestore db;

    public FirestoreExample() {
        db = FirebaseFirestore.getInstance();
    }

    public void crearUsuarioConPedido() {
        // 1. Crear usuario
        Map<String, Object> usuario = new HashMap<>();
        usuario.put("nombre", "Carlos");
        usuario.put("edad", 30);
        usuario.put("email", "carlos@example.com");

        DocumentReference usuarioRef = db.collection("usuarios").document("userId123");
        usuarioRef.set(usuario)
            .addOnSuccessListener(aVoid -> {
                Log.d("Firestore", "Usuario creado/actualizado con userId123");

                // 2. Añadir un pedido en la subcolección "pedidos"
                Map<String, Object> pedido = new HashMap<>();
                pedido.put("fecha", new Date());
                pedido.put("total", 59.99);
                pedido.put("estado", "Enviado");

                usuarioRef.collection("pedidos").add(pedido)
                    .addOnSuccessListener(docRef -> {
                        Log.d("Firestore", "Pedido creado con ID: " + docRef.getId());
                    })
                    .addOnFailureListener(e -> {
                        Log.e("Firestore", "Error al crear el pedido", e);
                    });

            })
            .addOnFailureListener(e -> {
                Log.e("Firestore", "Error al crear/actualizar usuario", e);
            });
    }
}
EXPLICACIóN
  1. Creamos o actualizamos un documento en la colección usuarios con ID userId123.
  2. Añadimos un documento en la subcolección pedidos.
  3. Comprobamos los listeners de éxito o error para mostrar mensajes de log.
