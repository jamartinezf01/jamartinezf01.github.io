Reglas de Seguridad

Firestore proporciona un sistema de reglas de seguridad basadas en expresiones para controlar el acceso a la base de datos. Estas reglas se configuran desde Firebase Console y determinan qué usuarios pueden leer o escribir en la base de datos, asegurando la privacidad y la integridad de los datos. través de este punto, aprenderemos a:

✅ Configurar reglas básicas (read y write) para permitir o restringir el acceso.
✅ Usar autenticación (request.auth) para restringir el acceso solo a usuarios autenticados.
✅ Definir reglas avanzadas con condiciones basadas en los datos del documento (resource).
✅ Proteger datos sensibles para que solo el usuario dueño del documento pueda acceder a su información.
✅ Evitar accesos no autorizados utilizando validaciones y restricciones.

1. Configuración básica de reglas en Firestore​

Las reglas de seguridad se configuran en la sección Firestore Database → Rules de Firebase Console.

Por defecto, Firestore no permite accesos públicos y requiere configuración. Una regla básica que permite acceso total a la base de datos es:

rules_version = '2';
service cloud.firestore {
  match /databases/{database}/documents {
    match /{document=**} {
      allow read, write: if true;
    }
  }
}
Cuidado:
  • Esta regla permite a cualquier usuario acceder y modificar datos, lo que no es seguro para aplicaciones en producción.

2. Reglas de acceso solo para usuarios autenticados (request.auth)​

Para permitir que solo usuarios autenticados puedan leer y escribir datos:

rules_version = '2';
service cloud.firestore {
  match /databases/{database}/documents {
    match /{document=**} {
      allow read, write: if request.auth != null;
    }
  }
}
Explicación:
  • request.auth != null verifica si el usuario está autenticado con Firebase Authentication.
  • Solo los usuarios registrados podrán acceder a la base de datos.

3. Reglas de seguridad para colecciones específicas​

Podemos definir reglas para restringir accesos a colecciones o documentos específicos.
Ejemplo: Permitir solo lectura en la colección "peliculas", pero sin escritura:

rules_version = '2';
service cloud.firestore {
  match /databases/{database}/documents {
    match /peliculas/{peliculaId} {
      allow read: if true;
      allow write: if false;
    }
  }
}
Explicación:
  • allow read: if true; → Cualquier usuario puede leer los documentos de "peliculas".
  • allow write: if false; → Nadie puede escribir en la colección.

4. Permitir que cada usuario acceda solo a sus propios datos​

Si cada usuario tiene un documento con su información dentro de la colección usuarios, podemos restringir el acceso solo al dueño del documento.

rules_version = '2';
service cloud.firestore {
  match /databases/{database}/documents {
    match /usuarios/{userId} {
      allow read, write: if request.auth != null && request.auth.uid == userId;
    }
  }
}
Explicación:
  • request.auth.uid == userId permite acceso solo si el usuario autenticado es el dueño del documento.
  • Se asume que los documentos de "usuarios" tienen como ID el mismo uid del usuario en Firebase Authentication.

5. Validación de datos antes de escribir en la base de datos​

Podemos definir reglas para asegurarnos de que los datos ingresados sean válidos.
Ejemplo: Restringir la creación de películas donde el año de estreno no sea mayor a 1900.

rules_version = '2';
service cloud.firestore {
  match /databases/{database}/documents {
    match /peliculas/{peliculaId} {
      allow write: if request.auth != null && request.resource.data.año > 1900;
    }
  }
}
Explicación:
  • request.resource.data.año > 1900 evita que se registren películas con años inválidos.
  • request.auth != null asegura que solo usuarios autenticados puedan escribir datos.

6. Evitar modificaciones de campos específicos​

Podemos permitir que los usuarios editen solo ciertos campos de un documento.
Ejemplo: Permitir que los usuarios actualicen solo la "reseña", pero no el "titulo" de una película.

rules_version = '2';
service cloud.firestore {
  match /databases/{database}/documents {
    match /reseñas/{resenaId} {
      allow update: if request.resource.data.keys().hasOnly(["reseña"]);
    }
  }
}
Explicación:
  • request.resource.data.keys().hasOnly(["reseña"]) verifica que el usuario solo intente modificar el campo "reseña".
  • Si intenta cambiar otro campo, la operación será rechazada.

7. Protección avanzada con condiciones en los datos (resource)​

Podemos aplicar reglas más complejas que verifiquen los valores de los documentos antes de conceder acceso.
Ejemplo: Permitir escribir una película solo si no supera los 300 minutos de duración.

rules_version = '2';
service cloud.firestore {
  match /databases/{database}/documents {
    match /peliculas/{peliculaId} {
      allow write: if request.auth != null && request.resource.data.duracion <= 300;
    }
  }
}
Explicación:
  • request.resource.data.duracion <= 300 asegura que las películas no tengan una duración superior a 300 minutos.
  • Esto previene datos inválidos y mantiene la coherencia en la base de datos.

8. Auditoría: Permitir solo creación de documentos, pero no edición ni eliminación​

Si queremos que los usuarios puedan crear documentos pero no editarlos ni eliminarlos, usamos allow create:.

rules_version = '2';
service cloud.firestore {
  match /databases/{database}/documents {
    match /comentarios/{comentarioId} {
      allow create: if request.auth != null;
      allow update, delete: if false;
    }
  }
}
Explicación:
  • allow create: if request.auth != null; → Solo permite crear documentos si el usuario está autenticado.
  • allow update, delete: if false; → Impide que los usuarios modifiquen o eliminen documentos.

9. Cómo probar las reglas de seguridad​

Antes de implementar las reglas en producción, podemos probarlas con el simulador de reglas de Firebase:

  1. Ir a Firebase Console → Firestore Database → Rules.
  2. Hacer clic en Simular una consulta.
  3. Seleccionar la colección y documento a probar.
  4. Elegir la operación (read, write, etc.).
  5. Ejecutar la prueba y verificar si la consulta es aceptada o rechazada.
