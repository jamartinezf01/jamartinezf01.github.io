Operaciones CRUD

En esta sección se explican de forma detallada las operaciones CRUD en Firebase Realtime Database usando Android (Java). Veremos cómo crear, leer, actualizar y eliminar datos en distintos escenarios, así como buenas prácticas en cada paso.

1. Referencia Principal​

En Firebase Realtime Database, la Referencia Principal (a menudo vista en código como mDatabase) es el punto de entrada para interactuar con tu base de datos. A partir de esa referencia, puedes navegar a cualquier nodo (o rama) del árbol JSON de tu base de datos.

La referencia principal hace referencia a la raíz de la base de datos; es decir, el nivel superior del árbol de datos en Realtime Database. Se obtiene generalmente así:

Referencia principal: Una instancia de la base de datos a través de:

DatabaseReference mDatabase = FirebaseDatabase.getInstance().getReference();
info
  • FirebaseDatabase.getInstance() obtiene la instancia de FirebaseDatabase configurada para tu proyecto (basado en el google-services.json que incluiste).
  • getReference() devuelve la referencia a la raíz de tu base de datos, también conocida como root.

2. Create (Crear)​

Para crear datos en Firebase Realtime Database, se pueden usar principalmente dos métodos:

  1. setValue(): Sobrescribe completamente un nodo.
  2. push(): Genera una clave única basada en una marca de tiempo y un identificador aleatorio, ideal para listas (por ejemplo, “mensajes”, “anuncios”, etc.).

Creación Directa con Clave Específica​

Si ya tienes un identificador (por ejemplo, el UID de un usuario) y deseas guardar datos en la ruta usuarios/<uid>:

private void crearUsuarioConClave(String uid, String nombre, String email) {
    Usuario usuario = new Usuario(nombre, email);
    mDatabase.child("usuarios").child(uid).setValue(usuario)
        .addOnCompleteListener(task -> {
            if (task.isSuccessful()) {
                Log.d("Create", "Usuario creado con éxito");
            } else {
                Log.e("Create", "Error al crear usuario", task.getException());
            }
        });
}
Explicación
  • mDatabase.child("usuarios").child(uid) apunta al nodo usuarios/uid.
  • setValue(usuario) crea (o sobrescribe) todo el nodo con la información de usuario.
ventajas y desventajas
  • Ventaja: Controlas exactamente dónde se guardan los datos (por ejemplo, usuarios/uid).
  • Desventaja: Necesitas tener un ID único ya existente

Creación con Clave Automática (push())​

Cuando no necesitas un identificador específico o cuando deseas guardar múltiples elementos como en una “lista”, se recomienda push():

private void crearAnuncio(String titulo, String mensaje) {
    Anuncio anuncio = new Anuncio(titulo, mensaje);
    // Generamos una clave automática en la rama "anuncios"
    String key = mDatabase.child("anuncios").push().getKey();
    
    if (key != null) {
        mDatabase.child("anuncios").child(key).setValue(anuncio)
            .addOnCompleteListener(task -> {
                if (task.isSuccessful()) {
                    Log.d("Create", "Anuncio creado con éxito");
                } else {
                    Log.e("Create", "Error al crear anuncio", task.getException());
                }
            });
    }
}
Explicación
  • push() crea un nodo con un ID único que se compone de la fecha y un componente aleatorio (por ejemplo: -Mxyz12345abc).
  • Es ideal para datos que se almacenan en formato de lista (chats, publicaciones, comentarios, etc.).
ventajas y desventajas
  • Ventaja: Perfecto para crear nodos en forma de lista sin preocuparte por colisiones de IDs.
  • Desventaja: No controlas directamente el ID. Aunque, en la mayoría de casos, ¡eso es lo que se quiere!

3. Read (Leer)​

La lectura puede hacerse de dos maneras principales:

  1. Lectura puntual (Single Read): Se utiliza normalmente para cargar datos una sola vez, sin escuchar cambios futuros.
  2. Lectura en tiempo real (Realtime Listener): Se suscribe a cambios en la base de datos; cada vez que el nodo cambia, se obtiene una actualización inmediata.

Lectura Puntual: addListenerForSingleValueEvent(...)​

Se llama una sola vez cuando se obtienen los datos, y luego se detiene:

private void leerUsuarioUnaVez(String uid) {
    mDatabase.child("usuarios").child(uid)
        .addListenerForSingleValueEvent(new ValueEventListener() {
            @Override
            public void onDataChange(@NonNull DataSnapshot dataSnapshot) {
                // Este método se llama cuando se obtienen los datos por primera vez
                Usuario usuario = dataSnapshot.getValue(Usuario.class);
                if (usuario != null) {
                    Log.d("Read", "Nombre: " + usuario.getNombre() + 
                                  ", Email: " + usuario.getEmail());
                } else {
                    Log.d("Read", "No existe el usuario con UID: " + uid);
                }
            }

            @Override
            public void onCancelled(@NonNull DatabaseError error) {
                Log.e("Read", "Error al leer datos", error.toException());
            }
        });
}

Lectura en Tiempo Real: addValueEventListener(...)​

Permite suscribirse de forma permanente a los cambios de un nodo. Cualquier cambio (adición, modificación o eliminación) en dicho nodo dispara nuevamente el callback onDataChange.

private void suscribirseAAnuncios() {
    mDatabase.child("anuncios").addValueEventListener(new ValueEventListener() {
        @Override
        public void onDataChange(@NonNull DataSnapshot snapshot) {
            // Este método se llamará cada vez que haya cambios en "anuncios"
            List<Anuncio> listaAnuncios = new ArrayList<>();
            for (DataSnapshot ds : snapshot.getChildren()) {
                Anuncio anuncio = ds.getValue(Anuncio.class);
                if (anuncio != null) {
                    listaAnuncios.add(anuncio);
                }
            }
            // Aquí actualizas tu UI con la listaAnuncios recién obtenida
            Log.d("Read", "Anuncios actualizados: " + listaAnuncios.size());
        }

        @Override
        public void onCancelled(@NonNull DatabaseError error) {
            Log.e("Read", "Error en la lectura de anuncios", error.toException());
        }
    });
}
explicación
  • onDataChange() se ejecuta tanto cuando se añade un nuevo anuncio como cuando cambia o se elimina uno.
  • Si solo requieres un evento por cada hijo que se añada o se cambie, puedes usar ChildEventListener, aunque en muchos casos ValueEventListener basta.

4. Update (Actualizar)​

Para modificar datos en Realtime Database existen dos métodos principales:

  1. setValue(): Si ya se tiene la referencia exacta de un nodo. Reemplaza todo el nodo (sobrescribe) con la nueva información.
  2. updateChildren(): Permite actualizar parcialmente múltiples campos o nodos simultáneamente (muy útil si no quieres sobrescribir todo).

Sobrescribir con setValue()​

private void actualizarUsuarioCompleto(String uid, Usuario usuarioActualizado) {
    mDatabase.child("usuarios").child(uid).setValue(usuarioActualizado)
        .addOnCompleteListener(task -> {
            if (task.isSuccessful()) {
                Log.d("Update", "Usuario sobrescrito con éxito");
            } else {
                Log.e("Update", "Error al sobrescribir usuario", task.getException());
            }
        });
}
aviso

En este caso, si el nodo usuarios/<uid> tenía más propiedades de las que tiene Usuario, se perderán al sobrescribir.

Actualización Parcial con updateChildren()​

Es especialmente útil cuando solo deseas modificar algunos campos sin perder información existente. Se emplea un Map<String, Object> con los campos a actualizar:

private void actualizarCampoUsuario(String uid, String nuevoNombre) {
    Map<String, Object> actualizaciones = new HashMap<>();
    actualizaciones.put("nombre", nuevoNombre);

    // También podrías actualizar más campos: actualizaciones.put("email", "otro@example.com");

    mDatabase.child("usuarios").child(uid).updateChildren(actualizaciones)
        .addOnCompleteListener(task -> {
            if (task.isSuccessful()) {
                Log.d("Update", "Usuario actualizado parcialmente");
            } else {
                Log.e("Update", "Error al actualizar usuario", task.getException());
            }
        });
}
explicación

updateChildren() aplica solo los cambios especificados y mantiene intactos los demás campos.

Transacciones y Concurrencia​

Cuando varios usuarios pueden escribir simultáneamente en un mismo nodo y necesitas consistencia (por ejemplo, contadores o “likes”), conviene usar transacciones:

private void incrementarContadorAnuncio(String anuncioId) {
    mDatabase.child("anuncios").child(anuncioId).child("contadorVisitas")
        .runTransaction(new Transaction.Handler() {
            @NonNull
            @Override
            public Transaction.Result doTransaction(@NonNull MutableData currentData) {
                Integer valorActual = currentData.getValue(Integer.class);
                if (valorActual == null) {
                    currentData.setValue(1);
                } else {
                    currentData.setValue(valorActual + 1);
                }
                return Transaction.success(currentData);
            }

            @Override
            public void onComplete(@Nullable DatabaseError error, boolean committed,
                                   @Nullable DataSnapshot currentData) {
                if (error != null) {
                    Log.e("Transaction", "Error al actualizar visitas", error.toException());
                } else {
                    Log.d("Transaction", "Contador de visitas actualizado");
                }
            }
        });
}
explicación

runTransaction(...) garantiza que si el valor ha cambiado mientras un usuario escribía, se reintente la operación con el valor actualizado, evitando sobreescrituras no deseadas.

5. Delete (Eliminar)​

Para borrar datos de la base de datos, se usa removeValue() sobre la referencia al nodo que deseas eliminar.

Eliminar un Nodo Específico​

private void eliminarUsuario(String uid) {
    mDatabase.child("usuarios").child(uid).removeValue()
        .addOnCompleteListener(task -> {
            if (task.isSuccessful()) {
                Log.d("Delete", "Usuario eliminado");
            } else {
                Log.e("Delete", "Error al eliminar usuario", task.getException());
            }
        });
}
  • Si usuarios/uid no existe, no ocurre nada; no es un error, simplemente no hay nada que eliminar.

Eliminar Múltiples Nodos​

Si quieres eliminar varios nodos a la vez, puedes usar updateChildren() con null:

private void eliminarVariosUsuarios(List<String> uids) {
    Map<String, Object> updates = new HashMap<>();
    for (String uid : uids) {
        // Al establecer el valor a null, se elimina esa ruta
        updates.put("/usuarios/" + uid, null);
    }
    mDatabase.updateChildren(updates)
        .addOnCompleteListener(task -> {
            if (task.isSuccessful()) {
                Log.d("Delete", "Usuarios eliminados correctamente");
            } else {
                Log.e("Delete", "Error al eliminar usuarios", task.getException());
            }
        });
}

Buenas Prácticas en CRUD​

Sigue estas prácticas
  1. Diseña la Estructura de Datos para facilitar las operaciones:
    • Evita anidar demasiados niveles.
    • Piensa en la distribución de datos para minimizar lecturas innecesarias.
  2. Reglas de Seguridad: Asegúrate de configurar reglas adecuadas para que solo los usuarios o roles deseados puedan crear, leer, actualizar o eliminar datos.
  3. Validación: Emplea updateChildren() para cambios parciales y evita sobrescribir información relevante accidentalmente.
  4. Transacciones: Son imprescindibles cuando necesitas consistencia en operaciones concurrentes (ej. contadores, votos, inventarios).
  5. Manejo de Errores: Siempre revisa onCompleteListener, onDataChange y onCancelled para notificar al usuario o registrar logs en caso de falla.
  6. Uso de Listeners:
    • Para cargas únicas, usa addListenerForSingleValueEvent.
    • Para datos en tiempo real, addValueEventListener o ChildEventListener.
  7. Performance y Costos: Evita leer grandes ramas de la base de datos innecesariamente. Aplica consultas y paginación si es preciso (usando orderByChild, startAt, limitToFirst, etc.).
