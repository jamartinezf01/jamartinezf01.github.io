Ejemplo Caso Práctico

La Liga Pokémon sigue necesitando un formato XML para gestionar la información de sus entrenadores, Pokémon y gimnasios, pero ahora se requiere que el documento sea validado mediante un DTD que incluya distintas restricciones, notaciones y entidades. El objetivo es que la estructura del XML y sus atributos cumplan ciertos criterios fijos, de modo que la información esté más controlada.

1. Enunciado​

  1. El elemento raíz se denominará <liga_pokemon> y deberá contar con:

    • Un atributo obligatorio fechaActualizacion (que contenga una fecha).
    • Un atributo obligatorio region, cuyo valor debe restringirse a un listado fijo (por ejemplo, "Kanto", "Johto", "Hoenn" o "Sinnoh").
    • Otros atributos opcionales que puedan ser de utilidad (por ejemplo, version, idioma).

  2. Sección de Entrenadores

    • Cada <entrenador> tendrá:
      • Un atributo id (obligatorio), con un código único (por ejemplo, E001, E002...).
      • Elementos obligatorios como <nombre> y <medallas> (número de medallas).
      • Un elemento opcional <favoritos>, que contenga referencias a Pokémon (por ejemplo, <pokemonRef>Pikachu</pokemonRef>).
    • Además, se podría agregar un atributo para indicar si el entrenador está activo en la liga (por ejemplo, activo="sí" o activo="no") y restringir estos valores en el DTD.

  3. Sección de Pokémon

    • Cada <pokemon> tendrá:
      • Un atributo idPokedex (obligatorio), que identifica de manera única al Pokémon (por ejemplo, 025 para Pikachu).
      • Un atributo legendario con valores restringidos a "sí" o "no".
      • Elementos obligatorios: <nombre> y <tipoPrincipal> (cuyo contenido podría restringirse en el DTD a "Fuego", "Agua", "Planta", "Eléctrico", "Volador", etc.).
      • Elementos opcionales: <tipoSecundario>, <nivel> y <habilidades> (con varios <movimiento>).
      • Un elemento <entrenadorRef> que indique a qué entrenador pertenece (por ejemplo, E001).
      • Un elemento <evoluciones> opcional, con uno o más <evolucionaA> (cada uno indicando el nombre de la evolución y la condición).

  4. Sección de Gimnasios

    • Cada <gimnasio> incluirá:
      • Un <nombre> y una <ciudad> obligatorios.
      • Un <tipoEspecialidad> restringido a un listado fijo (por ejemplo, "Agua", "Fuego", "Roca", "Eléctrico", etc.).
      • Opcionalmente, un <lider> con <nombre> y <medallaOtorgada>.
    • Se puede incluir un atributo para indicar si el gimnasio está abierto="sí" o abierto="no".

  5. Sección de Movimientos (opcional)

    • Cada <movimiento> tendrá:
      • Un <nombre> (por ejemplo, “Placaje”).
      • Un <tipo> (también restringido al mismo conjunto de tipos que <tipoPrincipal>).
      • Una <potencia> numérica.
      • Un <descripcion> opcional.

  6. Notations y Entidades

    • Se solicita incluir alguna NOTATION para manejar, por ejemplo, una imagen o logo relacionado con la liga, y/o unparsed entities.
    • Se podría declarar una entidad para símbolos especiales (por ejemplo, &pokecoin;) que represente la moneda usada en el mundo Pokémon.

  7. Datos de Ejemplo

    • Debe haber al menos dos entrenadores, tres o más Pokémon repartidos entre ellos, uno o dos gimnasios, y algún ejemplo de movimiento.
    • Se recomienda incluir valores que hagan uso de las enumeraciones y atributos mencionados (region, legendario, tipoEspecialidad, etc.).

Con estas restricciones, el DTD deberá definir la estructura (elementos, secuencias, opciones) y los atributos con valores fijos, enumeraciones, notaciones y entidades. Así, la Liga Pokémon contará con un fichero XML totalmente controlado, que podrá ser validado rigurosamente.

2. Posible solución DTD​

A continuación, se muestra un DTD de ejemplo que valida el XML propuesto para la Liga Pokémon, incluyendo distintas restricciones, enumeraciones, notaciones y entidades. El objetivo es ilustrar cómo estructurar un DTD que cubra la mayoría de los elementos y atributos descritos en el caso práctico modificado.

Guarda este contenido en un fichero con extensión .dtd (por ejemplo, liga_pokemon.dtd):

<!ELEMENT liga_pokemon (logo?, entrenadores, pokedex, gimnasios, movimientos?)>
<!ATTLIST liga_pokemon
    fechaActualizacion CDATA #REQUIRED
    region (Kanto | Johto | Hoenn | Sinnoh) #REQUIRED
    version CDATA #IMPLIED
    idioma CDATA #IMPLIED
>

<!-- Ejemplo de notaciones para imágenes -->
<!NOTATION JPG SYSTEM "image/jpeg">
<!NOTATION PNG SYSTEM "image/png">

<!-- Entidad de ejemplo para una moneda ficticia en el mundo Pokémon -->
<!ENTITY pokecoin "₱">

<!ELEMENT logo EMPTY>
<!ATTLIST logo
    nombreArchivo CDATA #REQUIRED
    formato (JPG | PNG) #REQUIRED
>

<!ELEMENT entrenadores (entrenador+)>
<!ELEMENT entrenador (nombre, medallas, favoritos?)>
<!ATTLIST entrenador
    id ID #REQUIRED
    activo (sí | no) "sí"
>

<!ELEMENT nombre (#PCDATA)>
<!ELEMENT medallas (#PCDATA)>

<!ELEMENT favoritos (pokemonRef+)>
<!ELEMENT pokemonRef (#PCDATA)>

<!ELEMENT pokedex (pokemon+)>
<!ELEMENT pokemon (nombre, 
                   tipoPrincipal, 
                   tipoSecundario?, 
                   nivel?, 
                   habilidades?, 
                   entrenadorRef, 
                   evoluciones?)>
<!ATTLIST pokemon
    idPokedex CDATA #REQUIRED
    legendario (sí | no) "no"
>

<!ELEMENT tipoPrincipal (#PCDATA)>
<!ELEMENT tipoSecundario (#PCDATA)>
<!ELEMENT nivel (#PCDATA)>

<!ELEMENT habilidades (movHabilidad+)>
<!ELEMENT movHabilidad (#PCDATA)>

<!ELEMENT entrenadorRef (#PCDATA)>

<!ELEMENT evoluciones (evolucionaA+)>
<!ELEMENT evolucionaA EMPTY>
<!ATTLIST evolucionaA
    nombre CDATA #REQUIRED
    condicion CDATA #IMPLIED
>

<!ELEMENT gimnasios (gimnasio+)>
<!ELEMENT gimnasio (nombre, ciudad, tipoEspecialidad, lider?)>
<!ATTLIST gimnasio
    abierto (sí | no) "sí"
>

<!ELEMENT ciudad (#PCDATA)>
<!ELEMENT tipoEspecialidad (#PCDATA)>

<!ELEMENT lider (nombre, medallaOtorgada)>
<!ELEMENT medallaOtorgada (#PCDATA)>

<!-- 
  Sección de movimientos (opcional).
  Cambiamos el nombre "movimiento" aquí para evitar conflicto
  con <movHabilidad> en <habilidades>, 
  pero en este DTD ejemplo asumimos que son elementos distintos.
-->
<!ELEMENT movimientos (movimiento+)>
<!ELEMENT movimiento (nombre, tipo, potencia, descripcion?)>

<!ELEMENT tipo (#PCDATA)>
<!ELEMENT potencia (#PCDATA)>
<!ELEMENT descripcion (#PCDATA)>
info

  1. Raíz <liga_pokemon>

    • fechaActualizacion es obligatorio (#REQUIRED).
    • region se restringe a (Kanto | Johto | Hoenn | Sinnoh).
    • version y idioma se declaran como opcionales (#IMPLIED).

  2. Notations

    • <!NOTATION JPG SYSTEM "image/jpeg"> y <!NOTATION PNG SYSTEM "image/png"> permiten manejar referencias a archivos no analizados (imágenes), como las del <logo>.

  3. Entidad pokecoin

    • Declarada como <!ENTITY pokecoin "₱">, de modo que en el XML podríamos usar &pokecoin; para representar esa moneda especial.

  4. Elemento <logo>

    • Definido como EMPTY (sin contenido), con atributos nombreArchivo y formato restringido a (JPG | PNG).

  5. Sección <entrenadores> y <entrenador>

    • entrenadores requiere uno o más (+) entrenadores.
    • Cada <entrenador> contiene <nombre> y <medallas> obligatorios, y <favoritos> opcional.
    • Atributo id se define como ID #REQUIRED, lo cual hace único ese identificador en el documento.
    • Atributo activo se restringe a (sí | no), con valor por defecto "sí".

  6. Sección <pokedex>

    • Agrupa uno o más <pokemon>.
    • Cada <pokemon> tiene el atributo idPokedex obligatorio y legendario="sí|no" con valor por defecto "no".
    • Su contenido exige ciertos elementos en orden:
      • <nombre> y <tipoPrincipal> obligatorios.
      • <tipoSecundario> y <nivel> opcionales (?).
      • <habilidades> opcional, <entrenadorRef> obligatorio, <evoluciones> opcional.

  7. Sección <gimnasios>

    • Múltiples <gimnasio>.
    • Cada <gimnasio> tiene un atributo abierto (sí/no) con valor por defecto "sí", y un contenido de <nombre>, <ciudad>, <tipoEspecialidad> y un <lider> opcional.

  8. Sección <movimientos> (opcional en la raíz)

    • Contiene uno o más (+) <movimiento>.
    • Cada <movimiento> se define con <nombre>, <tipo>, <potencia> y un <descripcion> opcional (?).

  9. Contenido de texto

    • Elementos como <nombre>, <medallas>, <pokemonRef>, <movHabilidad>, etc., se definen con (#PCDATA) para permitir texto interno.

3. XML válido (muy resumido)​

<?xml version="1.0" encoding="UTF-8"?>
<liga_pokemon fechaActualizacion="2025-12-31" region="Kanto" version="1.0" idioma="ES">
    
    <!-- Ejemplo de referencia a una posible entidad o archivo externo (no analizado) -->
    <logo nombreArchivo="liga_pokemon_logo.jpg" formato="JPG"/>
    
    <!-- Sección de Entrenadores -->
    <entrenadores>
        <entrenador id="E001" activo="sí">
            <nombre>Ash Ketchum</nombre>
            <medallas>6</medallas>
            <favoritos>
                <pokemonRef>Pikachu</pokemonRef>
                <pokemonRef>Charizard</pokemonRef>
            </favoritos>
        </entrenador>
        
        <entrenador id="E002" activo="no">
            <nombre>Misty</nombre>
            <medallas>4</medallas>
            <!-- Sin lista de favoritos -->
        </entrenador>
    </entrenadores>

    <!-- Sección de Pokémon -->
    <pokedex>
        <pokemon idPokedex="025" legendario="no">
            <nombre>Pikachu</nombre>
            <tipoPrincipal>Eléctrico</tipoPrincipal>
            <nivel>32</nivel>
            <habilidades>
                <movimiento>Impactrueno</movimiento>
                <movimiento>Onda Trueno</movimiento>
            </habilidades>
            <entrenadorRef>E001</entrenadorRef>
            <evoluciones>
                <evolucionaA nombre="Raichu" condicion="Piedra Trueno"/>
            </evoluciones>
        </pokemon>

        <pokemon idPokedex="006" legendario="no">
            <nombre>Charizard</nombre>
            <tipoPrincipal>Fuego</tipoPrincipal>
            <tipoSecundario>Volador</tipoSecundario>
            <nivel>40</nivel>
            <habilidades>
                <movimiento>Lanzallamas</movimiento>
            </habilidades>
            <entrenadorRef>E001</entrenadorRef>
        </pokemon>

        <pokemon idPokedex="054" legendario="no">
            <nombre>Psyduck</nombre>
            <tipoPrincipal>Agua</tipoPrincipal>
            <entrenadorRef>E002</entrenadorRef>
            <!-- Sin evoluciones, nivel, habilidades o segundo tipo -->
        </pokemon>
    </pokedex>

    <!-- Sección de Gimnasios -->
    <gimnasios>
        <gimnasio abierto="sí">
            <nombre>Gimnasio Celeste</nombre>
            <ciudad>Ciudad Celeste</ciudad>
            <tipoEspecialidad>Agua</tipoEspecialidad>
            <lider>
                <nombre>Misty</nombre>
                <medallaOtorgada>Medalla Cascada</medallaOtorgada>
            </lider>
        </gimnasio>

        <gimnasio abierto="no">
            <nombre>Gimnasio Plateado</nombre>
            <ciudad>Ciudad Plateada</ciudad>
            <tipoEspecialidad>Roca</tipoEspecialidad>
            <lider>
                <nombre>Brock</nombre>
                <medallaOtorgada>Medalla Roca</medallaOtorgada>
            </lider>
        </gimnasio>
    </gimnasios>

    <!-- Sección de Movimientos (opcional) -->
    <movimientos>
        <movimiento>
            <nombre>Placaje</nombre>
            <tipo>Normal</tipo>
            <potencia>40</potencia>
            <!-- Descripción opcional -->
            <descripcion>Un ataque físico básico.</descripcion>
        </movimiento>
        <movimiento>
            <nombre>Llamarada</nombre>
            <tipo>Fuego</tipo>
            <potencia>110</potencia>
        </movimiento>
    </movimientos>

</liga_pokemon>

Si este XML cumple la estructura declarada en el DTD, se considerará válido. Si agregamos elementos o atributos distintos a los permitidos, o incumplimos las enumeraciones, la validación fallará, evidenciando la utilidad del DTD para garantizar la calidad de los datos.
