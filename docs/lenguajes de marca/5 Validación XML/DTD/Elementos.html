Elementos

Las declaraciones de elementos (<!ELEMENT ...>) son el núcleo que define la estructura de un documento XML dentro de un DTD. Con ellas se controlan:

  • Qué elementos pueden aparecer. Cuáles son las etiquetas permitidas.
  • Cuál es su contenido (texto, subelementos, vacío…).
  • En qué orden deben aparecer, y Cuántas veces es posible usarlas.

Esto garantiza que los documentos XML que se ajusten a estas reglas sean consistentes y “hablen” el mismo lenguaje estructural, resultando en datos más fiables y fácil de procesar en cualquier aplicación o entorno.

1. Sintaxis básica​

La forma general de declarar un elemento en un DTD es:

<!ELEMENT nombre_elemento tipo_de_contenido>

Donde nombre_elemento es la etiqueta que aparecerá en el XML, y tipo_de_contenido describe qué puede contener.

2. Tipos de contenido​

Dentro de un elemento nos podemos encontrar con el siguiente contenido:

  • (#PCDATA): Texto sin subelementos.
  • EMPTY: Elemento vacío (sin contenido).
  • ANY: Puede contener texto, subelementos, o ambos.
  • Listado de subelementos (en paréntesis), con posibles operadores de secuencia (,), elección (|) y cardinalidad (*, +, ?).

Texto (#PCDATA)​

(#PCDATA) se refiere a “character data”, es decir, texto simple. Un elemento que solo contenga texto (y no subelementos) se declara así:

<!ELEMENT titulo (#PCDATA)>

Con esta declaración, <titulo>Algo de texto</titulo> es válido, y no se admiten subelementos dentro de titulo.

Por ejemplo​

Dado el XML...

<?xml version="1.0" encoding="UTF-8"?>
<!DOCTYPE noticia SYSTEM "noticia.dtd">
<noticia>
    <titulo>Nueva expansión de videojuego</titulo>
    <contenido>Se ha anunciado el lanzamiento de la próxima expansión con nuevas misiones y escenarios.</contenido>
</noticia>

Para validarlo con las siguientes condiciones...

  • El elemento raíz es <noticia>.
  • Dentro, obligatoriamente encontramos <titulo> y luego <contenido>.
  • Cada uno solo puede contener texto. deberemos emplear el DTD...
<!ELEMENT noticia (titulo, contenido)>
<!ELEMENT titulo (#PCDATA)>
<!ELEMENT contenido (#PCDATA)>
info
  • noticia contiene dos subelementos en ese orden: titulo y contenido.
  • Tanto titulo como contenido son puro texto ( (#PCDATA) ).

Vacío (EMPTY)​

Indica que el elemento no puede contener texto ni subelementos. Se utiliza, por ejemplo, en etiquetas “autocontenidas” que gestionan todo a través de atributos:

<!ELEMENT logo EMPTY>
info

El correspondiente XML podría ser <logo src="imagen.png"/>, sin ningún contenido dentro de <logo></logo>.

Cualquiera (ANY)​

Permite que el elemento contenga cualquier tipo de contenido, sea texto, subelementos o ambas cosas. Su uso se considera menos restrictivo, pues no impone reglas claras:

<!ELEMENT caja ANY>

Aquí, caja puede contener texto, elementos y atributos (estos últimos se declararían mediante ATTLIST).

Por ejemplo​

Un XML que válido para el anterior DTD sería...

<?xml version="1.0" encoding="UTF-8"?>
<!DOCTYPE caja SYSTEM "caja.dtd">
<caja>
    <contenido>
        <texto>Esto es un ejemplo de contenido dentro de caja</texto>
    </contenido>
    Aquí también meto texto libre.
</caja>
aviso
  • El DTD no pone límites: puedes tener subelementos (<contenido>) y texto suelto, mezclar, etc.
  • Se pierde mucho control de validación, por lo que ANY debe usarse con precaución.

Elementos hijos​

Si un elemento contiene uno o varios subelementos específicos, se listan dentro de paréntesis.

Por ejemplo:​

<!ELEMENT videojuego (titulo, genero, plataforma)>
info

En este caso, cada videojuego debe llevar exactamente tres subelementos (en ese orden): <titulo>, <genero> y <plataforma>.

3. Operadores de secuencia y de elección​

Los selectores de secuencia y selectores de elección en la declaración de elementos de un DTD (dentro de la sintaxis <!ELEMENT ...>) sirven para definir la organización de los subelementos que puede tener un elemento “padre”.

Selector de secuencia (,)​

Cuando se separan subelementos con comas, se indica que deben aparecer en el orden exacto especificado.

Ejemplo de uso en DTD​

<!ELEMENT persona (nombre, apellido)>

En este ejemplo, el elemento <persona> debe contener primero <nombre> y después <apellido> (en ese orden, y sin subelementos intercalados).

info
  • Garantiza un orden estricto: Ideal cuando la lógica de negocio o de datos exija que los subelementos aparezcan consecutivamente (por ejemplo, “primero el título, luego la fecha, luego el contenido”).
  • Validez estructural: Un documento XML que ponga <apellido> antes de <nombre> se consideraría inválido según la DTD.

Selector de elección (|)​

Cuando se separan subelementos con una barra vertical, se indica que puede aparecer uno u otro elemento, pero no ambos simultáneamente en esa posición.

Ejemplo de uso en DTD​

<!ELEMENT mensaje (sms | email)>

En este caso, <mensaje> tiene que contener un único subelemento que puede ser <sms> o <email>.

info
  • Flexibilidad de formato: Sirve para describir situaciones en las que un mismo tipo de elemento “padre” puede llevar opciones de estructura (por ejemplo, un solo método de contacto que puede ser un sms o un email, pero no ambos en la misma posición).
  • Rechazo de mezclas indebidas: Evita que un documento XML mezcle estructuras que no se esperan juntas.

4. Operadores de cardinalidad​

Los operadores de cardinalidad se utilizan para indicar cuántas veces puede (o debe) aparecer un subelemento dentro de un elemento “padre”. De este modo, se controla más finamente la estructura de un documento XML. Los principales operadores de cardinalidad son los siguientes...

  • *: 0 o más veces.
  • +: 1 o más veces.
  • ?: 0 o 1 vez.
  • (sin símbolo): exactamente 1 vez.

Por ejemplo:​

<!ELEMENT playlist (cancion+)>
info
  • (cancion+) implica que playlist tendrá una o varias etiquetas <cancion>.
Otro ejemplo:​
<!ELEMENT usuario (perfil?, experiencia*, nombre)>
info
  • perfil?: 0 o 1 perfil.
  • experiencia*: 0 o más experiencia.
  • nombre (sin símbolo): exactamente 1 nombre.
  • El orden será perfil (opcional), seguido de todas las experiencia que queramos, y finalmente nombre.

5. Ejemplos combinados​

Un elemento con secuencia y cardinalidades​

<!ELEMENT torneo (jugador+, equipo*, resumen?)>
info
  • jugador+: uno o más jugador.
  • equipo*: cero o más equipo.
  • resumen?: cero o uno resumen.
  • La secuencia indica que deben aparecer en ese orden: primero todos los jugadores, luego todos los equipos, y opcionalmente un resumen.

Un elemento con elección y cardinalidad​

<!ELEMENT notificacion (sms | mensajeapp)+>
info
  • La barra vertical | significa que cada subelemento puede ser sms o mensajeapp.
  • El operador + implica que dentro de <notificacion> habrá al menos uno y puede haber varios, en cualquier combinación.
  • El siguiente ejemplo sería válido...
<notificacion>
    <sms>...</sms>
    <mensajeapp>...</mensajeapp>
</notificacion>

Secuencia de elementos en una lista de opciones​

Podemos establecer secuencias estrictas de elementos como una parte de un lista de opciones...

<!ELEMENT notificacion ((pais, ciudad) | codigo_postal)+>

Los siguientes XML serían válidos para este DTD

<notificacion>
    <pais>...</pais>
    <ciudad>...</ciudad>
</notificacion>
<notificacion>
    <codigo_postal>...</codigo_postal>
</notificacion>
<notificacion>
    <pais>...</pais>
    <ciudad>...</ciudad>
    <codigo_postal>...</codigo_postal>
</notificacion>

Contenido mixto con #PCDATA en una lista de opciones​

En vez de emplear ANY, para permitir contenido míxto de manera más controlada podemos introducir #PCDATA enuna lista de opciones...

<!ELEMENT notificacion (#PCDATA | codigo | id)*>

6. Buenas prácticas al declarar elementos​

buenas prácticas
  1. Mantén la estructura clara: Emplea secuencias solo si el orden es realmente importante; si no, valora otras opciones (o un ANY en casos muy abiertos).
  2. Controla cuidadosamente la cardinalidad: Evita declarar todo con * o ANY, ya que se pierde la fuerza de la validación.
  3. Nombres descriptivos: Usa nombres de elementos (producto, titulo, precio) que indiquen claramente su propósito.
  4. Evita abusar de ANY: Aunque es flexible, no aporta comprobaciones y deja al XML casi sin restricciones.
  5. Comenta tu DTD: Si trabajas en equipo, es útil explicar el porqué de la secuencia o la opción en un comentario.
