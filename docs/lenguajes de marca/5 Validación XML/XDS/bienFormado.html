XSD bien formado

Un XSD bien formado es aquel que cumple con las reglas de sintaxis y estructura propias de XML y que, además, sigue los lineamientos de la especificación XML Schema. Esto garantiza que el documento XSD sea legible, fácil de mantener y que las herramientas (validadores, editores, etc.) puedan interpretarlo correctamente.

1. Requisitos para un XSD bien formado​

I. Cumplir con la sintaxis básica de XML

  • Todo elemento debe tener su etiqueta de apertura y cierre correspondientes.
  • Debe existir un único elemento raíz (en este caso, <xs:schema>).
  • Las entidades especiales (<, >, &) deben escaparse adecuadamente dentro del contenido cuando sea necesario.

II. Incluir la declaración del namespace de XML Schema

  • Generalmente se indica con xmlns:xs="http://www.w3.org/2001/XMLSchema".
  • Todas las construcciones de XSD (ej. <xs:element>, <xs:complexType>, <xs:simpleType>, etc.) deben usar ese prefijo xs o cualquier otro que hayamos definido para el namespace de XML Schema.

III. Declarar el targetNamespace (si corresponde)

  • Si el esquema define elementos y tipos para un espacio de nombres concreto, usar targetNamespace="...".
  • De no requerirse un namespace específico (caso poco frecuente en entornos reales), no se utiliza targetNamespace.

IV. Opciones de configuración adicionales

  • elementFormDefault="qualified": Para que los elementos declarados en el esquema pertenezcan al namespace.
  • attributeFormDefault="unqualified" (opcional): Suele usarse para que los atributos no estén obligados a tener namespace.

V. Estructura interna clara

  • Incluir comentarios (opcional, pero recomendable) para describir secciones, elementos o tipos.
  • Separar la declaración de tipos, elementos y grupos de atributos si el XSD es extenso.

2. Estructura principal de un XSD​

La estructura general de un XSD suele ser la siguiente:

<xs:schema 
    xmlns:xs="http://www.w3.org/2001/XMLSchema"
    targetNamespace="http://www.ejemplo.org/miesquema"
    xmlns="http://www.ejemplo.org/miesquema"
    elementFormDefault="qualified"
    attributeFormDefault="unqualified">

    <!-- Sección de import o include (si aplica) -->
    <!-- xs:import namespace="..." schemaLocation="..." -->
    <!-- xs:include schemaLocation="..." -->

    <!-- Definiciones de tipos simples -->
    <!-- <xs:simpleType name="..."> ... </xs:simpleType> -->

    <!-- Definiciones de tipos complejos -->
    <!-- <xs:complexType name="..."> ... </xs:complexType> -->

    <!-- Definiciones de grupos de elementos o atributos (opcional) -->
    <!-- <xs:group name="..."> ... </xs:group> -->
    <!-- <xs:attributeGroup name="..."> ... </xs:attributeGroup> -->

    <!-- Declaraciones de elementos raíz u otros elementos -->
    <!-- <xs:element name="..."> ... </xs:element> -->

</xs:schema>
En este esquema:
  • <xs:schema>: Elemento raíz que indica que se trata de un XSD.
  • xmlns:xs="...": Namespace oficial de la W3C para las construcciones del lenguaje de esquemas.
  • targetNamespace: URL que define el espacio de nombres al cual pertenecen los elementos y tipos definidos en este XSD.
  • xmlns="...": Hace que, por defecto, los elementos que se definan se vinculen a ese namespace (sin necesidad de usar prefijos).
  • elementFormDefault y attributeFormDefault: Especifican cómo se aplican los namespaces a los elementos y atributos.

3. Ejemplo de XSD bien formado​

A continuación, un ejemplo que define esquemáticamente una lista de productos tecnológicos:

<xs:schema 
    xmlns:xs="http://www.w3.org/2001/XMLSchema"
    targetNamespace="http://www.ejemplo.org/productos"
    xmlns="http://www.ejemplo.org/productos"
    elementFormDefault="qualified"
    attributeFormDefault="unqualified">

    <!-- Tipo simple para restringir el nombre del producto -->
    <xs:simpleType name="tipoNombreProducto">
        <xs:restriction base="xs:string">
            <xs:minLength value="1"/>
            <xs:maxLength value="50"/>
        </xs:restriction>
    </xs:simpleType>

    <!-- Tipo complejo para un producto -->
    <xs:complexType name="tipoProducto">
        <xs:sequence>
            <xs:element name="Nombre" type="tipoNombreProducto"/>
            <xs:element name="Categoria" type="xs:string"/>
            <xs:element name="Precio" type="xs:decimal"/>
        </xs:sequence>
        <xs:attribute name="id" type="xs:int" use="required"/>
    </xs:complexType>

    <!-- Elemento raíz: Lista de productos -->
    <xs:element name="CatalogoProductos">
        <xs:complexType>
            <xs:sequence>
                <xs:element name="Producto" type="tipoProducto" maxOccurs="unbounded"/>
            </xs:sequence>
        </xs:complexType>
    </xs:element>

</xs:schema>

Análisis del ejemplo​

  1. <xs:simpleType> (tipoNombreProducto):
    Un tipo restringido basado en xs:string con longitud mínima de 1 y máxima de 50.
  2. <xs:complexType> (tipoProducto):
    • Contiene tres elementos (Nombre, Categoria, Precio) y un atributo id.
    • El atributo id es obligatorio (use="required").
  3. Elemento raíz (CatalogoProductos):
    • Define un contenedor de varios productos (maxOccurs="unbounded" indica que puede haber muchos).
    • Para validar un XML usando este XSD, el documento debería tener un elemento <CatalogoProductos> que incluya múltiples <Producto>.

Este XSD es bien formado porque:

  • Sigue la sintaxis XML adecuadamente.
  • Usa correctamente el namespace de XML Schema (xmlns:xs).
  • Declara de forma clara el targetNamespace.
  • Incluye las definiciones de tipos simples y complejos, además de un elemento raíz.
  • Respeta la apertura y cierre correcto de cada etiqueta.

4. Validación de la estructura​

Para confirmar que un XSD está correctamente escrito, podemos:

  1. Utilizar un editor o IDE que soporte validación de XSD (por ejemplo, Visual Studio Code con extensiones específicas, XMLSpy, Oxygen XML Editor, etc.).
  2. Usar herramientas en línea (W3C XML Schema Validator, por ejemplo) que permitan subir el .xsd y verifiquen la estructura.

Además de estar bien formado (a nivel de sintaxis XML), se puede comprobar que las definiciones dentro del XSD sean válidas a nivel de esquema: que no haya duplicación de nombres, referencias inválidas, etc.
